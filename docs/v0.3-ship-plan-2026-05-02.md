# v0.3 Ship Plan — Port LOCOMO Judge Into engram-bench

**Filed:** 2026-05-02 (early morning, after long debugging session)
**Status:** Plan — not yet executed
**Owner:** potato
**Suggested next issue ID:** ISS-100

---

## TL;DR

Stop using `cogmembench/benchmarks/locomo/engram_adapter.py` as engram's LOCOMO ship-gate measurement engine. Port the LLM-as-judge pipeline (~135 lines of Python) into `crates/engram-bench` as ~200 lines of Rust. Estimated effort: **1–2 days**, not the 1 week I initially mis-quoted.

After this lands, engram measures its own LOCOMO J-score with no dependency on cogmembench. Cogmembench goes back to being a neutral multi-system benchmark (its actual purpose).

---

## Strategic Context (recap from a week ago)

Two-legged strategy for engram v0.3 release:

- **Leg 1 — Defense:** beat published baselines on mainstream benchmarks
  - LOCOMO ≥ 68.5 % (parity with mem0) — `GOAL-5.1`, P0 ship gate
  - LongMemEval ≥ v0.2 baseline + 15 pp — `GOAL-5.3`, P0 ship gate
- **Leg 2 — Offense:** publish our own benchmark (cogmembench, 6-level cognitive axes)

**Both legs need LOCOMO**, but in different roles:
- Leg 1: LOCOMO is the *target* (must hit ≥ 68.5 %)
- Leg 2: LOCOMO is *one of several baselines* cogmembench tests across systems

The two legs **must use different code paths**. Right now they don't, which is why we're filing this plan.

## Current State (verified, not assumed)

### What's broken

1. **Leg 1 has no measurement path of its own.** `engram-bench` crate has skeletons for `drivers/locomo.rs` + `scorers/locomo.rs` (normalised exact match), but **no LLM-as-judge scorer**. So leg 1 cannot produce a J-score number on its own.

2. **Leg 1 is parasitic on Leg 2.** To get any J-score number at all, we run `cogmembench/benchmarks/locomo/engram_adapter.py`. Cogmembench is supposed to be a neutral multi-system benchmark; using it for engram's own ship gate pollutes that neutrality.

3. **The parasitic path is currently broken anyway.** RUN-0016 hit a `reindex --ns` CLI flag mismatch (cogmembench adapter calls a flag the current engram CLI removed). Patching the adapter again deepens the pollution.

4. **No clean J-score number exists for v0.3.** Recent runs:
   - RUN-0012: substrate corruption (ISS-098)
   - RUN-0013: missing meta side-channel
   - RUN-0015: didn't finish
   - RUN-0016: reindex flag broken
   No trustworthy ship-gate measurement has ever been recorded.

### What's working (foundation we can reuse)

In `crates/engram-bench/`:
- `src/drivers/locomo.rs` — LOCOMO data loading + retrieve loop already exists
- `src/scorers/locomo.rs` — normalised exact-match scorer already exists
- `Cargo.toml` — `reqwest = "0.12"` already a transitive dependency via `engramai`

In `cogmembench/benchmarks/`:
- `common/evaluator.py` (~50 lines) — `judge_answer` + `generate_answer` prompts
- `common/llm.py` (~80 lines) — Claude API client over OAuth/HTTP
- `locomo/runner.py` (~291 lines) — full driver loop with checkpointing
- `locomo/engram_adapter.py` — engram-specific adapter (will be retired from leg 1's path)

**Total reference Python: ~135 lines of judging logic + ~80 lines of LLM client = 215 lines.** This is what we're porting.

### Step 1 — `llm_client.rs` (~2h)

Minimal Anthropic Messages API client. **No OAuth.** Read `ANTHROPIC_API_KEY` from env.

```rust
pub struct LlmClient {
    client: reqwest::blocking::Client,
    api_key: String,
    model: String,           // default: "claude-haiku-4-5"
}

impl LlmClient {
    pub fn from_env() -> Result<Self, LlmError>;
    pub fn call(&self, prompt: &str, max_tokens: u32) -> Result<String, LlmError>;
}
```

- Use `reqwest::blocking` (already in tree) — driver loop is sync, no need to fight Tokio
- Surface errors explicitly (per INV1 — "never silent degrade")
- Retry policy: 3 attempts with exponential backoff on 5xx / network errors
- Log token usage per call to a `Vec<TokenStat>` so we can write cost report (`G3` measurement hook)

**Acceptance:** smoke test — call with `"Reply Yes"` prompt, assert response startswith "Yes".

### Step 2 — `scorers/locomo_judge.rs` (~3h)

Direct port of `benchmarks/common/evaluator.py:judge_answer`:

```rust
pub fn judge_answer(
    client: &LlmClient,
    question: &str,
    predicted: &str,
    gold: &str,
) -> Result<JudgeOutcome, LlmError> {
    if predicted.starts_with("LLM_ERROR") {
        return Ok(JudgeOutcome { is_correct: false, raw: "error".into() });
    }
    let prompt = format!(
        "You are an evaluator. Determine if the predicted answer is semantically \
         equivalent to the gold answer for the given question.\n\
         Consider an answer correct if it conveys the same key information, \
         even if worded differently.\n\
         For \"I don't know\" predictions against unanswerable questions, mark as correct.\n\n\
         Question: {q}\n\
         Gold Answer: {g}\n\
         Predicted Answer: {p}\n\n\
         Is the predicted answer correct? Reply with exactly \"Yes\" or \"No\".",
        q = question, g = gold, p = predicted,
    );
    let resp = client.call(&prompt, 50)?;
    let is_correct = resp.trim().to_lowercase().starts_with("yes");
    Ok(JudgeOutcome { is_correct, raw: resp })
}

pub fn generate_answer(
    client: &LlmClient,
    question: &str,
    context: &str,
) -> Result<String, LlmError> {
    let prompt = format!(
        "Based on the following conversation memories, answer the question concisely.\n\
         If the answer is not in the memories, say \"I don't know.\"\n\n\
         MEMORIES:\n{ctx}\n\n\
         QUESTION: {q}\n\n\
         Answer (be concise, just the key fact):",
        ctx = context, q = question,
    );
    client.call(&prompt, 300)
}
```

**Prompts copied byte-for-byte** from cogmembench so cross-validation against existing cogmembench runs is exact. Any wording change later is a separate decision.

### Step 3 — `drivers/locomo.rs` modification (~3h)

Current driver: load LOCOMO → for each question → retrieve memories → pass to scorer.

New flow:
```
load LOCOMO
for each question:
    retrieved = retrieve(question)        # existing
    context  = format_memories(retrieved) # existing
    answer   = generate_answer(client, question, context)  # NEW
    outcome  = judge_answer(client, question, answer, gold) # NEW
    record { question, answer, gold, outcome.is_correct, outcome.raw, retrieved }
write run report (JSON) → .gid/eval-runs/RUN-NNNN/
```

Also port `compute_evidence_recall` (~10 lines, pure substring match — no LLM).

**Checkpointing:** mirror `runner.py` — write JSONL after each question so a crashed run resumes cleanly. (Important for 199-question full runs.)

### Step 4 — Smoke test (~1h)

- Run 25 questions on conv-26
- Compare each `(question, answer, judge_verdict)` tuple against the same questions run through cogmembench's `engram_adapter.py`
- **Acceptance:** verdict mismatch ≤ 1/25 (one borderline disagreement is acceptable; more means port has a bug)

### Step 5 — Full run (~next day)

- Run all 199 LOCOMO questions
- Record J-score, evidence_recall, latency, token cost
- Write to `.gid/eval-runs/RUN-NNNN/results.json` + `summary.md`
- This is the **first credible v0.3 ship-gate measurement**

## What This Plan Does NOT Do

Deliberately out of scope for ISS-100:

1. **Does not consolidate the two-DB architecture** (`<stem>.db` + `<stem>.graph.db`).
   - That's a separate architectural debt — see "Open Architectural Questions" below.
   - ISS-100 succeeds whether or not we ever merge the two files.

2. **Does not modify cogmembench.**
   - The `reindex --ns` bug in `engram_adapter.py` stays unfixed *for engram's purposes*.
   - cogmembench team (us, but in leg-2 mode) can fix it on their own schedule for cogmembench's own runs.

3. **Does not port LongMemEval judging.**
   - LongMemEval is a separate dataset and ship gate (`GOAL-5.3`).
   - Same pattern will apply once LOCOMO is proven, but defer until needed.

4. **Does not change the ship-gate threshold.**
   - 68.5 % stays as the target. We measure first, then decide if reality forces revision.

## Risk & Fallback

**Risk:** Rust port produces different J-scores than cogmembench because of subtle prompt or LLM-call differences.

**Mitigation:** Step 4 cross-validation. If verdict mismatch > 1/25, debug before proceeding.

**Fallback if port fails:** Revert to fixing `engram_adapter.py:reindex --ns` for one run only, get an interim number, then come back and finish the port. Worst case adds 1 day, doesn't change the strategic direction.

## Open Architectural Questions (NOT for this plan)

These came up during tonight's discussion. **Logging here so they don't get lost.** Each deserves its own issue once v0.3 ships.

1. **Two-DB split is a patch, not a design.**
   - `<stem>.graph.db` separation was never discussed in `DESIGN-v0.3.md`.
   - Came in via early implementation choice, codified by ISS-046's "FK on for same-file, off for separate" workaround.
   - SQLite cannot enforce cross-file foreign keys → entities/edges have no DB-level integrity guarantee.
   - This is the root cause of ISS-058 split-brain class of bugs.
   - **Suggested follow-up issue:** "Reconcile two-file implementation with one-substrate design (DESIGN §0/G2/§6)."

2. **Cogmembench boundary hygiene.**
   - As long as engram doesn't depend on cogmembench (post-ISS-100), cogmembench can evolve as a neutral benchmark.
   - But its `engram_adapter.py` still has accumulated engram-specific patches from earlier debugging.
   - **Suggested follow-up issue:** "Audit cogmembench/benchmarks/locomo/engram_adapter.py — remove engram-team-only debugging artefacts."

3. **engram-bench → engram-bench-cli → cogmembench overlap.**
   - Three layers of "things that run benchmarks." Roles unclear.
   - Defer until after v0.3 ships and we see how the layers actually get used.
