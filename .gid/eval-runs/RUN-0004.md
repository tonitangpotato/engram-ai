## Issue / context

ISS-067 (Hybrid fallback) was filed at end of B1. RUN-0003 closed the Hybrid story (sub-plans genuinely return empty, not a mapping bug). B2 picks one of the three open directions left at end of RUN-0003 and digs.

**Direction chosen:** investigate the **5 Factual misses** (q2, q5, q8, q19, q20 — same plan, same outcome=ok, but gold not in top-5). Reasoning: most actionable inside autopilot scope (no API change, no new flag), and it sits squarely inside the "what is recall actually capped at?" question Phase B is meant to answer.

## Goal

Decide whether the 5 Factual misses are:
- (a) a retrieval-side ranking problem (gold memory exists, just not in top-5), or
- (b) an ingestion-side gap (gold memory was never persisted), or
- (c) a gold-annotation noise issue (LoCoMo evidence label disagrees with what the question really asks).

Three different fixes, three different priorities. Need ground truth.

## Hypothesis

H1 (most likely going in): retrieval-side — embeddings are off for these specific phrasings ("identity", "relationship status"), gold memory is in DB but ranks below distractors. Plan: rerank or query-rewriting.

H2 (less likely): ingestion-side — turns dropped during extraction. Plan: instrument extractor.

## Method

### Preflight discovery (the namespace twist)

First retrieval invocation used `--ns conv26` (matching RUN-0002 / RUN-0003). New behavior:

```
WARN engramai::retrieval] graph_query: namespace "conv26" not found
   (no memories, no entities) — returning empty result set
```

Every query: empty. hit@5 = 0/25.

`SELECT DISTINCT namespace FROM memories` on the substrate → namespace is `locomo-conv26-iss058`, not `conv26`. RUN-0002 / RUN-0003 ran against the same substrate and reported real hits despite passing `--ns conv26`. That can only mean the pre-ISS-064 retrieval path was **not actually scoping by namespace** — it returned data regardless. ISS-064 ("namespace mismatch silently returns empty") was filed against exactly this footgun; the workspace edit to `crates/engramai/src/retrieval/api.rs` (uncommitted, present during this run) added the fast-fail that turned the silent footgun into a loud one.

So:
- The namespace fix is doing its job.
- RUN-0002 and RUN-0003's headline number is real but was produced under "leaky" scoping. Re-running with the *correct* namespace (`locomo-conv26-iss058`) is the apples-to-apples comparison.

This is an unrelated finding worth recording because it changes the meaning of every prior `--ns conv26` retrieval run in the test log.

### Ingestion-gap details

Substrate has 31 memories total, distinct sources from sessions 1-3:

```
D1: 3, 5, 7, 9, 11, 14, 16              (7 of 18 turns)
D2: 1, 2, 3, 5, 7, 8, 10, 12, 14        (9 of 15 turns)
D3: 1, 2, 3, 4, 5, 6, 7, 10, 11, 13, 14, 17  (12 of 17 turns)
```

≈ 28/50 turns (56%) survive ingestion. `quarantine` table count = 0 — these turns are not quarantined, they're skipped silently before any quarantine codepath runs (or merged into an adjacent turn under a different `source`).

Speaker-filter hypothesis fails: D1:14 (Melanie, kept) vs D1:12 (Melanie, dropped). Both are content-rich. Real driver is probably extractor success/failure on the LLM call for each turn; needs instrumentation to confirm.

### Retrieval-rank details (q5, q8)

q5 "What is Caroline's identity?" — gold memory `d7aa6167` content is "Caroline found transgender stories inspiring and felt happy and thankful for the support received". The word "identity" doesn't appear; "transgender" does, but never paired with "Caroline is …". Embedding similarity loses to D3:3 / D2:10 (both have explicit "Caroline" mentions in life-event contexts). This is a **query-rewrite or expansion** problem, not a ranking-loss problem.

q8 "What is Caroline's relationship status?" — gold memory `62abc42e` content is "Caroline plans to become a single parent and create a family for children in need". "Single parent" is implicitly a relationship status, but the embedding doesn't bridge that. Same shape: query rewriting or schema-aware retrieval would help; reranking inside the existing top-50 won't.

## Observations

- **Confirmed**: the LoCoMo conv-26 s1-3 hit@5 ceiling at 56.0% is *real and stable* — two runs apart, same substrate, identical numbers. Stagnation criterion in autopilot rules met (Δ between consecutive RUNs < 0.02).
- **Surprised**: ingestion gap is the larger fraction of Factual misses (3 of 5), not retrieval ranking. Going in I assumed (H1) most misses were rank-loss. Wrong by 60%.
- **Confirmed (incidentally)**: ISS-064 fix is correctly surfacing what was previously a silent footgun. RUN-0002/0003 numbers are valid but should be re-noted in the test log: "ran against `conv26` namespace which silently fell back to unscoped retrieval; equivalent to running against `locomo-conv26-iss058` directly under the new fast-fail."
- **Surprised**: gold annotation noise is real for q2 — LoCoMo lists D1:12 as evidence for "When did Melanie paint a sunrise?", but D1:12 is "You'd be a great counselor!" with no painting/date content. The actual answer ("last year") sits in D1:14, which the script *did* return in top-5. So q2's "miss" is partly an evaluation artifact, not a system failure. Doesn't change the headline number (still scores as miss because dia_id check is exact), but worth flagging if we ever do per-question error analysis at scale.

## Decision

**file-issue + exit Phase B.**

Filed **ISS-068**: "Pipeline extractor drops conversational turns referenced as gold evidence (LoCoMo conv-26 D1:12, D2:15, D3:16)". P1, retrieval/ingestion. Has repro, hypothesis (Five Whys), and acceptance criteria. Recommends instrumenting `PipelineRecordProcessor::process_record` to log per-turn outcome before any fix attempt — first job is observability, second is fix.

**Why exit:**
- Stop condition (b) met: hit@5 stagnant <0.02 across two consecutive RUNs (56.0% → 56.0%). Continuing in the retrieval/Hybrid plane will not move the needle — the bottleneck is one layer up, in ingestion.
- Continuing into ingestion bug-hunting is out of autopilot scope (requires re-ingest cycles which are expensive and need OpenAI quota). That's a daytime task.

The reflection (per autopilot B-EXIT spec) goes in STATUS / Phase B summary, not here.
