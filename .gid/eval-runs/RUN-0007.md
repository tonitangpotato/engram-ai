## Why this run exists

ISS-072 ("graph entity creation degrades to defaults") found that prior runs produced graph_entities where `kind` defaulted to `other("unknown")` for ~99% of rows because the LLM extractor's `subject_kind`/`object_kind` hints were dropped on the floor between extraction and persistence. A-clean closes that plumbing gap.

This run **must** be a fresh ingest (not substrate reuse) because `kind` is a column on `graph_entities`, baked in at write time. Querying yesterday's DB with today's binary would tell us nothing about today's binary.

## Goal (GOAL-2 acceptance)

Reduce `kind=other("unknown")` ratio to **≤30%** of `graph_entities` rows on the canonical LoCoMo conv-26 corpus (50 turns, sessions 1-3).

## Findings

### F1. GOAL-2 PASS — kind plumbing works end-to-end

The 99.07% → 14.97% drop is the cleanest before/after we've had on this issue. The remaining 28 `unknown` rows are LLM judgement calls (extractor genuinely couldn't decide), not plumbing failures — distinguishable by the `attributes.kind_source` column: rows tagged `Default` are LLM-uncertain, `TripleHint` are LLM-decided. This split is itself a small win for observability.

### F2. summary/importance are 100% empty — but NOT a regression

ISS-072 issue body listed `summary` and `importance` alongside `kind` as "degrades to defaults" symptoms, implying A-clean would fix all three. RUN-0007 shows it only fixed `kind`. Root cause traced via:

```
$ grep -nE "summary|importance" crates/engramai/src/triple_extractor.rs
(no matches)
```

The LLM extractor's prompt and output schema only ask for `subject`, `predicate`, `object`, `subject_kind_hint`, `object_kind_hint` (see `Triple` struct in `triple.rs`). Summary/importance are **never on the wire** — A-clean's plumbing has nothing to plumb for those fields. This is a **prompt/schema gap**, not a plumbing gap.

This is consistent with ISS-072's GOAL-2.b note that "enrichment fields are deferred" — but the issue body's framing of summary/importance as "degraded defaults" was misleading. They're not degraded; they were never extracted.

### F3. attributes column is provenance, not data

The 100% populated `attributes` is misleading at first glance. The schema is `{"kind_source": "TripleHint"|"Default"[, "subtype_hint": ...]}` — A-clean's own debugging breadcrumb. Real entity attributes (e.g., for a `person`: pronouns, role, relation; for an `event`: when, where) are not extracted at all. Same root cause as F2: not in the extractor schema.

## Acceptance verdict

- **GOAL-2 (kind ratio ≤ 30%)**: **PASS** ✅ — 14.97%
- **GOAL-2.b (enrichment fields)**: not in scope for this PR (deferred per issue) — RUN-0007 confirms `summary`/`importance` need separate work upstream of A-clean (extractor prompt expansion)
