## TL;DR

After ISS-023 repo consolidation, `engram/.gid/graph.db` contains only 8 nodes (ISS-001 feature + its 7 subtasks, dated 2026-04-19). The deprecated `engram-ai-rust/.gid/graph.yml` contains ~114 nodes representing the full pre-consolidation project history. Once ISS-028 Option A registers the ~14 active issues manually, the graph is functionally fine for new work — but it still lacks the pre-2026-04-19 historical context (old features, completed tasks, architectural nodes).

This ticket tracks **two possible approaches** to recovering that history. Both are "nice to have" — not required for current or near-term work.

## Context — What's Missing

`engram/.gid/graph.db` currently contains (after ISS-001 perf work):
- `iss-001` (feature), 7 child tasks (all done)
- 14 edges

`engram-ai-rust/.gid/graph.yml` (deprecated repo, preserved on disk) contains:
- ~114 nodes — a mix of feature nodes, task nodes, code nodes (files/classes/functions), and some historical ISS-xxx nodes that were registered before the split
- Last meaningful update: 2026-04-22 23:33 (pre-consolidation)
- Includes ISS-024 graph registration (per MEMORY.md note)

**What we lose without this:**
- Ability to run `gid_query_impact` on pre-consolidation code nodes
- Historical context for "when was this feature done?" queries
- Cross-references from old issue nodes to code nodes they touched
- Architecture visualization that includes the full codebase (not just new issues)

**What we do NOT lose:**
- Current work tracking (ISS-028 Option A handles this)
- Any canonical source of truth (issue .md files are canonical; the graph is derived view)
- Any ability to ship fixes / do reviews (designs and investigations are in .gid/issues/)

## Approaches

### Option B: Full YAML → DB Migration

Merge all ~114 nodes from `engram-ai-rust/.gid/graph.yml` into `engram/.gid/graph.db`.

**Steps:**
1. Parse old `graph.yml` (gid-core should have a yaml-reading path, verify)
2. Map yaml node/edge schema → db schema (should be 1:1 since both come from gid-core)
3. Dedup against existing 8 nodes (ISS-001 and its children are already in db)
4. Insert into db preserving all fields (metadata, tags, timestamps)
5. Verify: `gid_read --project engram` shows merged graph without conflicts

**Estimated effort:** 1–2 hours (dominated by schema mapping verification + dedup logic)
**Risk:** medium — any schema drift between yml-era and db-era gid-core could cause data loss. Must do dry-run first.
**Blocker:** none technical, but requires focused session — not for a tired end-of-day slot.

### Option C: Rebuild From `.gid/issues/` Folder

Scan `.gid/issues/ISS-*/` and auto-generate feature nodes + child tasks from the markdown structure.

**Steps:**
1. Write a one-off scanner (Rust bin or shell script) that:
   - Walks `.gid/issues/ISS-*`
   - For each issue, parses the `**Status:**` field, title, related/blocks/blocked-by links
   - Emits `gid_add_task` calls (or direct SQL) with `node_type=feature`, status extracted from markdown
   - For issues with `design.md` that contains `### 9. Tasks` or similar graph-YAML snippets, parse those for child task nodes
2. Run scanner, get generated nodes
3. Dedup vs existing db (ISS-001 subtree)
4. Commit

**Estimated effort:** 2–3 hours (scanner is ~200 LOC but needs test coverage)
**Risk:** low for issues (structure is consistent), higher for old code-level nodes (no source to regenerate from without `gid_extract`)
**Coverage:** this approach recovers issue-level history but NOT code-level nodes (files, classes, functions). For code-level nodes, we'd need to run `gid_extract src/` fresh on the monorepo — which gives current state, not historical state.

### Hybrid (Recommended if we ever do this)

1. Run `gid_extract src/` on engram monorepo → rebuild all current code-level nodes (fast, deterministic)
2. Run Option C scanner → rebuild all issue feature nodes from markdown
3. Skip Option B entirely — the old yaml's code nodes are stale anyway after consolidation, and its issue nodes will be regenerated by step 2

This gives us "graph matches current reality" rather than "graph contains historical snapshot", which is probably what we actually want.

## Decision Criteria — When to Do This

Do this work if/when:
- We want `gid_query_impact` to work for architectural planning
- We want graph-based LoCoMo / benchmark work (graph as input to recall features)
- We want `gid_visual` to produce a meaningful diagram for engram
- Someone other than potato needs to navigate the codebase via graph

Do NOT do this if:
- We're about to rewrite / restructure major parts of the codebase (rebuild later, cheaper)
- The immediate need is just issue tracking (ISS-028 Option A is sufficient)
- We're optimizing recall quality / LoCoMo score (unrelated path)
