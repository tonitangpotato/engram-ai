## 🔴 Critical (blocks correctness)

### FINDING-1: Bottleneck 1 描述与实际代码不符 — data loading 已经是 batch pattern

ISS 写道：
> `synthesis/cluster.rs` 的 `discover_clusters()` 为每条记忆单独查 DB

实际代码（`synthesis/cluster.rs:127-148`）：

```rust
// Hebbian links
let mut hebbian_map: HashMap<String, Vec<(String, f64)>> = HashMap::new();
for m in &candidates {
    if let Ok(links) = storage.get_hebbian_links_weighted(&m.id) {
        // ...builds HashMap...
    }
}

// Entity IDs per memory
let mut entity_map: HashMap<String, Vec<String>> = HashMap::new();
for m in &candidates {
    if let Ok(entities) = storage.get_entity_ids_for_memory(&m.id) {
        entity_map.insert(m.id.clone(), entities);
    }
}
```

这确实是 N 次 per-ID SQL 查询，ISS 的问题诊断是对的。但 ISS 的 "Before" 代码段里展示的是在 `for mem` 里逐条构建 `ClusterNode`，暗示 signal maps 也是在 node 构建循环里做的 —— 实际上 signal maps 的构建是在 **Step 2** 独立完成的，ClusterNode 构建是 **Step 3**，已经从 HashMap 取数据了。

**问题**：ISS 的 Fix 1 "Before" 代码是错的。`discover_clusters` 已经把查询和 node 构建分成了两步（Step 2 和 Step 3）。Fix 1 改的是 Step 2 的查询方式（N 次 per-ID → 2 次 bulk），不是把查询从 node 循环里移出来。

**Suggested fix：** 更正 Fix 1 的 "Before/After" 代码段，准确反映 Step 2 的代码。核心改动不变（bulk SQL 替代 per-ID SQL），但描述要准确。

### FINDING-2: VP-tree cosine distance 不满足 metric 条件 — 文档承认了但 mitigation 不够

ISS Notes 里写：
> cosine distance 严格来说不是完美的 metric（不满足 d(x,x)=0 当向量未归一化时），但 embedding 向量通常是归一化的

实际问题比这更严重。Cosine distance（1 - cosine_sim）在高维空间中 **不满足三角不等式**，即使向量归一化：

- 对归一化向量，cosine distance = `1 - dot(a,b)` = angular 差异的近似
- 真正的 angular distance（`arccos(dot(a,b)) / π`）满足三角不等式
- 但 `1 - cosine_sim` **不满足**。反例：三个单位向量 a, b, c 在某些角度组合下 `d(a,c) > d(a,b) + d(b,c)`

VP-tree 剪枝的正确性 **依赖三角不等式**。如果不满足，VP-tree 可能漏掉真正的近邻。

**Suggested fix：** 两个选项：
- **(a)** 改用 angular distance：`distance = arccos(clamp(dot(a,b), -1, 1)) / π`。这是严格的 metric，数值稳定。计算量多一个 `acos` 但每次查询只对 ~O(log n) 个节点算，可以接受。
- **(b)** 改用 Euclidean distance on L2-normalized vectors。对归一化向量 `||a - b||² = 2(1 - cos(a,b))`，所以 `||a - b|| = sqrt(2(1 - cos(a,b)))`。Euclidean distance 满足三角不等式。但需要先确认 nomic-embed-text 输出是否 L2-归一化。

推荐 **(b)**：Euclidean on normalized vectors，因为 nomic-embed-text 的输出 [通常是归一化的](https://huggingface.co/nomic-ai/nomic-embed-text-v1.5)，且 sqrt + 减法比 acos 快。同时在 VP-tree 构建前加一个 assertions 确认 L2 norm ≈ 1.0。

## 🟡 Important (should fix before implementing)

### FINDING-4: 漏掉了第三个 O(m²) 热点 — cohesion 计算和 find_centroid

ISS 只识别了两个瓶颈（data loading + k-NN graph），但实际有第三个：

1. `clustering.rs:339-347`：cohesion 计算是 **per-cluster O(m²)**

```rust
for (pi, &i) in member_indices.iter().enumerate() {
    for &j in &member_indices[(pi + 1)..] {
        let s = self.strategy.compute_weight(&nodes[i], &nodes[j]);
        // ...
    }
}
```

2. `synthesis/cluster.rs:213-226`：`find_centroid` 也是 per-cluster O(m²)

```rust
for (i, &node_a) in members.iter().enumerate() {
    for (j, &node_b) in members.iter().enumerate() {
        if i != j {
            sum += strategy.compute_weight(node_a, node_b);
        }
    }
}
```

如果最大 cluster 有 m=50 个成员，这每个 cluster 是 50×49/2 = 1,225 次 `compute_weight`。如果有 200 个 clusters，总共 ~245k 次。不是主瓶颈，但是 **同一个 compute_weight 函数**，如果不小心可能隐藏 bug（比如 Hebbian 的 HashMap 查找在这个阶段仍然有效吗？是有效的，因为 Hebbian links 存在 `ClusterNode.hebbian_links` 里而不是外部 map）。

**建议：** ISS 的 "What Does NOT Change" 表格应提到这两个 O(m²) 点，确认它们在 cluster 内部运行（m ≤ 50），不需要优化，但实现时不要意外破坏。

### FINDING-5: VP-tree k' (over-fetch factor) 缺乏质量保证分析

ISS 写 k'=3k 默认足够，但没有分析 **会漏掉什么样的边**。

关键问题：MultiSignal 权重 = 0.4×hebbian + 0.3×entity + 0.2×embedding + 0.1×temporal。

如果两个记忆 Hebbian weight = 0（没有 co-recall 历史）但 entity overlap 非常高（比如都提到 "RustClaw, engram, VP-tree"），它们的 MultiSignal 权重可能 ~0.3（entity 贡献），但 embedding 距离可能很远（语义完全不同，只是碰巧共享实体）。

VP-tree 用 embedding ANN 做候选剪枝 → 这些 entity-driven edges 可能被完全漏掉。Hebbian pairs 注入解决了 Hebbian 信号的问题，但 **entity-overlap pairs 没有注入策略**。

**Suggested fix：** 增加 entity 候选注入，类似 Hebbian 注入：
```
candidate_pairs = ANN_top_k'(embedding) ∪ hebbian_linked_pairs ∪ entity_co_occurring_pairs
```
其中 `entity_co_occurring_pairs` = 共享 ≥1 个 entity 的记忆对。这在 bulk load entity 数据后很容易构建（反向索引 entity_id → [memory_ids]，然后对每个 entity 的 memory 列表取两两配对）。

或者，如果 entity 数据太稀疏（大多数记忆没有 entity），可以在实际数据上先 profile 再决定。但 ISS 应该至少 **承认这个 gap**，而不是假设 ANN + Hebbian 就够了。

### FINDING-6: Fix 1 bulk 方法缺少 `candidate_ids` 过滤问题

当前 per-ID 查询代码在查 Hebbian 后有一个过滤：

```rust
if candidate_ids.contains(neighbor.as_str()) {
    hebbian_map.entry(m.id.clone()).or_default().push((neighbor, weight));
}
```

只保留 candidate 集合内的 Hebbian links。Bulk 方法如果返回 **全量** Hebbian links（包括非 candidate 的记忆），调用端需要过滤。

ISS 的 Fix 1 "After" 代码写的是 `all_hebbian.get(&mem.id).cloned().unwrap_or_default()` —— 没有做 candidate 过滤。这意味着 `ClusterNode.hebbian_links` 可能包含不在 candidates 里的 memory ID，导致 `MultiSignal::compute_weight` 里的 Hebbian lookup 有无关 entries（不影响正确性 —— 找不到 match 返回 0 —— 但浪费内存）。

**Suggested fix：** 要么 bulk 方法不变，调用端做过滤（现有代码已经做了，ISS 的 "After" 漏了）；要么 bulk 方法接受一个 `candidate_ids: &HashSet<&str>` 参数做 SQL WHERE IN 过滤。推荐前者 —— 简单，且 candidate 过滤已经是现有模式。

### FINDING-7: `EmbeddingOnly` 路径（compiler/discovery.rs）也有同样的 O(n²) 问题 — ISS 没有覆盖

`compiler/discovery.rs` 也调用 `InfomapClusterer::cluster()`，走 `EmbeddingOnly` 策略。它面对的同样是 13k+ 节点的 O(n²) k-NN 图构建。

ISS 只谈了 `synthesis/cluster.rs` 的 `MultiSignal` 场景。但 VP-tree 对 `EmbeddingOnly` 更有效（只有一个信号，ANN 精度最高，不需要任何注入策略）。

**Suggested fix：** 明确说明 Fix 2（VP-tree two-stage）在 `InfomapClusterer::cluster()` 内部实现，所以 **两条路径都自动受益**。ISS 应该在 scope 里提到 `compiler/discovery.rs` 也被覆盖了。

## 🟢 Minor (can fix during implementation)

### FINDING-8: VP-tree 实现建议不够详细 — query 算法缺描述

ISS 给了 `VpTree` 的结构体定义，但 `query_nearest` 的 pruning 逻辑没有写出来。VP-tree 查询的核心是：

```
if dist(query, vantage_point) - tau < threshold:
    search left subtree
if dist(query, vantage_point) + tau >= threshold:
    search right subtree
```

其中 `tau` 是当前已知第 k 近邻的距离。这个剪枝条件决定了查询性能。ISS 应该至少给出伪代码级别的描述，确保实现者不会写错剪枝条件。

### FINDING-10: "no new dependencies" 声明需要确认 — VP-tree 不需要 `ordered-float` 之类的？

VP-tree 需要比较距离值。`f64` 不实现 `Ord`（NaN）。实现 VP-tree 时需要处理 NaN 安全的排序。如果用 `partial_cmp + unwrap_or`（当前代码风格），无需新 crate。但应明确说明。
