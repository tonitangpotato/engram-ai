## FINDING-1 🔴 Critical — [Check #29] Design claims `Storage::connection(&self)` doesn't exist; it does

**§4** states:

> Required prereq: add `Storage::connection(&self) -> &rusqlite::Connection` (today only `connection_mut(&mut self)` exists).

**Ground truth** (`crates/engramai/src/storage.rs:300-302`):

```rust
/// Get a reference to the underlying database connection.
pub fn connection(&self) -> &Connection {
    &self.conn
}
```

The method already exists and is actively used (~15 callsites in tests and prod — `storage.connection().query_row(...)` etc.). The design's "one-line accessor" prereq is already satisfied.

**Impact:** The §6 Q2 open question ("Storage::connection(&self) accessor — safe to add?") is moot — it's already shipped and in use. This removes one of the four stated blockers. The design must be corrected to acknowledge the existing accessor and drop Q2 from the blocker list.

**Suggested fix:** In §4, replace "Required prereq: add `Storage::connection(&self)`…" with "Uses existing `Storage::connection(&self)` (storage.rs:300)." Remove Q2 from §6 or reclassify as "already answered."

## FINDING-2 🔴 Critical — [Check #29] Design claims `graph_store` field is always `None` / dead code; it isn't

**§1** states:

> …which is a lie: the field is always `None` (search confirms — every constructor path sets `graph_store: None`). The `'static` is unreachable…

**Ground truth** (`crates/engramai/src/memory.rs`):

- `with_pipeline_pool` (line 356): `self.graph_store = Some(Arc::clone(&store_arc));` — constructs a `SqliteGraphStore<'static>` via `Box::leak(Box::new(conn))`.
- `with_graph_store` (line 516): `self.graph_store = Some(Arc::new(Mutex::new(store)));` — same pattern.
- `with_graph_read(&self)` (line 555): actively borrows the stored graph store for read-only access through a closure + mutex guard pattern.

The `'static` is achieved deliberately via `Box::leak`. The field is NOT dead code — it's the primary mechanism for v0.3 retrieval plans to access the graph. The constructors (`Memory::new`, etc.) set it to `None`, but `with_pipeline_pool` and `with_graph_store` are builder methods that populate it later.

**Impact:** This fundamentally changes the problem statement. The design's §1 motivation ("fix dead code") is factually wrong. The *real* motivation (enabling `&self` graph reads without the mutex/Arc ceremony) is valid but needs reframing. More importantly, the refactor must account for the existing `Arc<Mutex<SqliteGraphStore<'static>>>` field and `with_graph_read` accessor — the design never mentions these and proposes `Memory::graph(&self)` as if it's greenfield.

**Suggested fix:** Rewrite §1 to accurately describe the existing `graph_store` field + `with_graph_read` mechanism. Add a migration note: does `Memory::graph(&self)` replace `with_graph_read`, coexist with it, or is the existing pattern kept for the `Arc<Mutex<...>>` case? This is a design gap, not just a doc error.

## FINDING-3 🟡 Important — [Check #29] Design §2 trait impl for `GraphStore` is misleading but accidentally correct

**§2** proposes:

```rust
impl<C: BorrowMut<Connection>> GraphStore for SqliteGraphStore<C>
```

The actual trait hierarchy in the codebase (`store.rs:603-604`):

```rust
pub trait GraphStore: GraphWrite {}
impl<T: GraphWrite + ?Sized> GraphStore for T {}
```

`GraphStore` has a **blanket impl** — anything implementing `GraphWrite` automatically gets `GraphStore`. So the design's explicit `impl GraphStore` line is dead code; it would conflict with the blanket impl.

**Why it works anyway:** The blanket impl means `SqliteGraphStore<C>` where `C: BorrowMut<Connection>` gets `GraphStore` for free once `GraphWrite` is implemented. No explicit impl needed. The design is accidentally correct in outcome but wrong in mechanism.

**Impact:** Low — the compiler will reject the redundant impl and the implementer will remove it. But the design should be precise about the trait topology since the `GraphStore: GraphWrite` (not `GraphRead + GraphWrite`) hierarchy is load-bearing for the split.

**Suggested fix:** Remove the explicit `impl GraphStore` line from §2. Add a note: "GraphStore is blanket-implemented for any `T: GraphWrite`, so only `GraphRead` and `GraphWrite` impls are needed."

## FINDING-4 🟡 Important — [Check #5/6] Design doesn't address existing `with_graph_read` / `Arc<Mutex<...>>` data flow

The codebase has two graph access paths that the design ignores:

1. **`Memory::graph_mut(&mut self)`** — addressed by design ✅
2. **`Memory::with_graph_read(&self, f: impl FnOnce(&dyn GraphRead) -> R)`** — NOT addressed ❌

The `with_graph_read` path uses `self.graph_store: Option<Arc<Mutex<SqliteGraphStore<'static>>>>` and hands out `&dyn GraphRead` through a closure to avoid leaking the mutex guard. This is the production read path for v0.3 retrieval plans.

The design proposes `Memory::graph(&self) -> SqliteGraphStoreReader<'_>` which borrows `self.storage.connection()`. But retrieval plans currently use `with_graph_read` which goes through a **separate leaked connection** in the `Arc<Mutex<...>>` field.

**Questions the design must answer:**
- Does `Memory::graph(&self)` replace `with_graph_read`? If so, what happens to callers?
- If they coexist, reads can happen on two different connections (storage's vs the leaked one) — is this intentional?
- The leaked connection in `graph_store` may have different WAL state than `storage.connection()`. Is read consistency guaranteed?

**Suggested fix:** Add a §4.1 addressing the relationship between `graph(&self)` and `with_graph_read`. Decide: deprecate `with_graph_read` and migrate callers, or keep both with documented semantics.

## FINDING-5 🟡 Important — [Check #14/32] Design vs issue naming inconsistency for the generic struct

The **issue** (`issue.md`) proposes:

```rust
pub struct SqliteGraphStoreT<C: Borrow<rusqlite::Connection>> { ... }
pub type SqliteGraphStore<'a> = SqliteGraphStoreT<&'a mut rusqlite::Connection>;
pub type SqliteGraphReader<'a> = SqliteGraphStoreT<&'a rusqlite::Connection>;
```

The **design** proposes:

```rust
pub struct SqliteGraphStore<C: Borrow<rusqlite::Connection>> { ... }
pub type SqliteGraphStoreOwned<'a> = SqliteGraphStore<&'a mut rusqlite::Connection>;
pub type SqliteGraphStoreReader<'a> = SqliteGraphStore<&'a rusqlite::Connection>;
```

These are fundamentally different strategies:
- **Issue:** Rename struct to `SqliteGraphStoreT`, keep `SqliteGraphStore<'a>` as a type alias → existing code with explicit `SqliteGraphStore<'a>` compiles unchanged (zero-diff for Group 2 too).
- **Design:** Keep struct name `SqliteGraphStore`, introduce `SqliteGraphStoreOwned<'a>` alias → Group 2 sites (27) must be renamed.

The issue's approach is **strictly more backward-compatible** (0 renames vs 27). The design doesn't acknowledge this alternative or explain why it chose the more disruptive path.

**Suggested fix:** Either adopt the issue's `SqliteGraphStoreT` naming (fewer changes) or add a trade-off note in §3 explaining why the design prefers the current naming despite requiring 27 renames.

## FINDING-8 🟢 Minor — [Check #4] Inconsistent alias naming: `Reader` vs `Owned` asymmetry

`SqliteGraphStoreOwned` (for `&mut Connection`) and `SqliteGraphStoreReader` (for `&Connection`). The naming is asymmetric: "Owned" describes the *ownership model* while "Reader" describes the *capability*. A `&mut Connection` borrow is not "owned" — it's a mutable borrow. Consistent naming would be either:
- Capability-based: `SqliteGraphStoreWriter` / `SqliteGraphStoreReader`
- Borrow-based: `SqliteGraphStoreMut` / `SqliteGraphStoreRef`

**Suggested fix:** Rename to `SqliteGraphStoreMut<'a>` / `SqliteGraphStoreRef<'a>`, or `Writer`/`Reader`. Minor — can be decided during implementation.
