## FINDING-1 🟢 Minor — [Check #4] Method name mismatch: `dispatch_sub_plan` vs actual `run`

**Section:** §4.1 header, §3 diagram

**Problem:** The design doc refers to the Hybrid sub-plan dispatch method as `HybridDispatchExecutor::dispatch_sub_plan` (§4.1 header: "Hybrid sub-plan dispatcher (`HybridDispatchExecutor::dispatch_sub_plan`, lines 640-647)"). The §3 diagram also references `HybridDispatchExecutor::dispatch_sub_plan(SubPlanKind::Abstract, ...)`.

The actual method in `orchestrator.rs:553` is `fn run(&mut self, kind: SubPlanKind) -> SubPlanResult` (implementing the `HybridSubPlanExecutor` trait). The issue.md similarly uses `dispatch_subplan`.

This is a documentation-only error (the code is correct), but anyone navigating from the design doc to the source will be confused by `grep dispatch_sub_plan` returning zero hits.

**Suggested fix:** Replace all references to `dispatch_sub_plan` / `dispatch_subplan` with `run` (the actual method name), or clarify that `run` is the trait method and "dispatch" is the conceptual role.

## FINDING-2 🟡 Important — [Check #30] Stale module comment constitutes undocumented drift

**Section:** §6 (Boundary), §4

**Problem:** The orchestrator module comment at `orchestrator.rs:499-505` still reads:

> **v0.3 collaborator slots are deferred.** The executor wires `Null*` implementations of every plan's recaller / resolver / store.

This was true pre-ISS-049 Phase 3 but is now false — real collaborators are wired via `PlanCollaborators`. The design doc's §6 boundary section carefully lists what ISS-059 does NOT touch, but doesn't flag this stale comment as a known pre-existing inaccuracy in the file it edits. Since ISS-059 modifies code ~140 lines below this comment in the same struct's `run` method, the stale comment is in the immediate neighborhood of the fix.

This isn't technical debt ISS-059 *introduces*, but it's debt that ISS-059 *walks past* — a reviewer reading the module comment will believe collaborators are still Null-stubbed, contradicting the fix's entire rationale.

**Suggested fix:** Add to §6 Boundary: "B-7. The module-level comment at `orchestrator.rs:499-505` describing `Null*` stubs as current behavior is stale post-ISS-049 Phase 3. Updating it is out of ISS-059 scope but should be filed as a cleanup." Alternatively, update the comment as part of ISS-059 since it's a 2-line edit in the same file.

## FINDING-3 🟡 Important — [Check #31] Dual-extraction of namespace is a latent fragility

**Section:** §3 (Propagation chain), Invariant I-2

**Problem:** The namespace is extracted from `query.namespace` at **two independent sites**:

1. `api.rs:409-411` — clones `query.namespace` into an owned `String`, threads it to adapter constructors (`HybridSeedRecaller`, `HybridAffectiveSeedRecaller`) for Factual/Episodic/Associative/Affective plans.
2. `orchestrator.rs:643-647 / 945` — reads `query.namespace.as_deref()` inside `execute_plan` for Abstract plan inputs.

The design doc acknowledges this in I-2 ("same string, same `None` behavior") but doesn't flag it as a fragility. Today both reads come from the same `GraphQuery` instance and are safe. But if any future refactor (a) modifies `query.namespace` between `api.rs` extraction and `execute_plan` entry, or (b) makes `dispatch()` consume/move `query.namespace`, the two sites silently diverge — one plan type queries one namespace, the rest query another, reproducing exactly the ISS-059 bug class.

This isn't a current bug — it's the *pattern* that caused ISS-059 in the first place (two sites constructing the same value independently). The design doc's §4.3 discusses why a helper was rejected for `AbstractPlanInputs`, but doesn't address the broader dual-extraction between `api.rs` and `orchestrator.rs`.

**Suggested fix:** Add to §8 (Failure modes): "F-4. Dual namespace extraction. `api.rs` clones `query.namespace` for adapter construction; `orchestrator.rs` re-reads `query.namespace` for `AbstractPlanInputs`. If a future refactor moves or mutates the query between these sites, the namespace can diverge silently. FW-4's `AbstractPlanInputs::new(query, budget)` constructor would partially address this; a single `namespace: &str` threaded through `execute_plan` (alongside the existing `namespace_str` parameter) would fully collapse the two extraction sites."

## FINDING-5 🟡 Important — [Check #33] AC #3 claim masks pre-existing test failures without qualification

**Section:** §7.1 (Verification — Done)

**Problem:** AC #3 states: "Existing tests do not regress. `cargo test -p engramai` shows three pre-existing failures unrelated to namespace propagation (knowledge_compile contract drift × 2, ISS-047 ledger gap × 1)."

The design claims these failures are "unrelated to namespace propagation" but provides no evidence for that claim beyond an external reference to `memory/2026-04-29.md 02:50 entry`. A reviewer cannot verify the "unrelated" assertion from the design doc alone. If any of these three failures *is* namespace-related (e.g., if `knowledge_compile` processes data under a namespace), the fix could be masking a regression.

This isn't a shortcut (check #31) — the failures genuinely appear pre-existing. But "three failures exist, trust me they're unrelated" without inline evidence violates the design doc's own standard of verifiability.

**Suggested fix:** Add a one-liner per failure in §7.1 AC #3 explaining *why* each is unrelated. E.g.: "`knowledge_compile` tests do not exercise `AbstractPlanInputs` or `GraphQuery.namespace`; they test compile-stage entity extraction (ISS-047 scope). The `ledger gap` test predates the namespace field entirely (filed 2026-04-2X)."

## FINDING-6 🟢 Minor — [Check #20] Minor prose inaccuracy: `.into()` vs `.to_string()`

**Section:** §3 (Invariant I-2)

**Problem:** Invariant I-2 states the `api.rs` expression is `unwrap_or_else(|| "default".into())`. The actual code at `api.rs:412` is `unwrap_or_else(|| "default".to_string())`. Semantically identical (`Into<String>` for `&str` calls `to_string()`), but a reviewer grepping for the exact expression won't find it. Trivial.

**Suggested fix:** Change `.into()` to `.to_string()` in §3 I-2.
