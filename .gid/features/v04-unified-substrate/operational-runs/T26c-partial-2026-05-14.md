## Hypotheses for the 41% failure rate

None verified. Listed for follow-up in ISS-129 instrumentation work.

- **A. Anthropic burst-pattern rate limit.** Sustained ~34 req/min over 7
  hours may have tripped a per-hour or per-day quota that the 173-second
  sample never hit. Direct curl with the same OAuth token returns HTTP 200
  in 0.55s, so the credential is healthy.
- **B. Token throughput limit.** Some memories are long; cumulative input
  tokens over hours may have hit a TPM ceiling.
- **C. Transient network errors.** Process held the same DNS-resolved
  connection for hours; intermittent TCP resets would each consume one
  retry budget.
- **D. Driver giving up after a streak.** The outer loop terminates if too
  many consecutive memories hit max_retries. Worth re-reading the source
  to confirm.

ISS-128 fix (persist failed memory_ids + capture HTTP error body) is the
prerequisite for any of these to be verified.
