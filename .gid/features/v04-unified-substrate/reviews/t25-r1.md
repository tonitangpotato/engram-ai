The 2 Important findings are both about **Phase B / Phase C parity for the `attributes` JSON**. T25 silently diverges from Phase B T16 in three ways: it includes `synthesis_timestamp` (Phase B doesn't), it OMITS keys when source values are NULL (Phase B emits `null`), and the FK guard runs as N+1 round-trips when the outer JOIN could fold it.

Most of the "decisions documented in §8.4" are correct — the legacy.id pass-through is exactly right; the confidence pass-through policy is correct and well-documented. But the implicit `weight = 1.0` / `activation = 0.0` (via column defaults rather than explicit params in the helper) is a load-bearing parity assumption that should be made explicit.

## Findings

### FINDING-1 — Important — Phase B/C attributes JSON divergence
**Severity**: Important
**Location**:
  - `crates/engramai/src/substrate/backfill.rs:2310-2356` (T25 attribute construction)
  - `crates/engramai/src/storage.rs:4575-4593` (Phase B T16 attribute construction)

**What's wrong**:
The `attributes` JSON written by T25 (backfill) and T16 (Phase B dual-write) diverges in three observable ways for the SAME logical row:

| key | Phase B T16 emits | T25 backfill emits | diverges when |
|---|---|---|---|
| `synthesis_timestamp` | NOT present | always present (RFC3339 string) | always |
| `gate_scores` (legacy NULL) | `null` | key omitted | legacy gate_scores IS NULL or empty |
| `source_original_importance` (legacy NULL) | `null` | key omitted | legacy column IS NULL |

This means: an operator inspecting `edges WHERE edge_kind='provenance'` cannot tell which rows came from backfill vs Phase B by content shape, AND a JSON-extract query `WHERE json_extract(attributes, '$.synthesis_timestamp') IS NOT NULL` will partition the table by writer, NOT by data. The `id` pass-through invariant (T25's headline contract) is correct because re-emission would collide on PK — but the attributes blob would be the LAST writer's shape, silently overwriting either side.

Wait — actually it's `INSERT OR IGNORE`. So a re-emission DOESN'T overwrite. Which means: if backfill ran first and Phase B's writer fires later for the same row, the Phase B row is silently dropped. The attributes blob a reader sees is whichever side won the race. This is the desired idempotency BUT it also means the two writers must produce byte-equivalent attributes JSON for the parity invariant to hold under any race order. They currently don't.

**Why it matters**:
- T27 (verification report) is going to assert row-count parity and likely spot-check content parity. The three keys above will fail that spot-check.
- Operators writing forensic queries on `attributes` will get different results depending on which writer happened to run first.
- The "matching id" property in verifier I5 (§6) is necessary but not sufficient — operators reasonably expect "matching id implies matching content".
- **Two roots are possible, and the right one depends on what the spec is:**
  - (a) Phase B is correct; T25 should also emit `synthesis_timestamp=NULL`-removed (i.e. drop synthesis_timestamp; emit `null` for null inputs).
  - (b) T25 is correct; Phase B should add `synthesis_timestamp` and omit-NULL.
  - (c) Both are wrong; a third canonical shape exists.

§4.5 / §3.3 don't pin the attributes content schema for provenance edges. So neither writer is strictly violating the spec — they're filling a gap.

**Suggested fix**:
Pick (b): T25's behavior is the better one (`synthesis_timestamp` is forensically valuable; omit-NULL is JSON convention and reduces blob size). Apply two changes:

1. Add `synthesis_timestamp` to Phase B T16's `json_object` call. One-line spec amendment + one-line code amendment.
2. Change Phase B T16 to use `json_object` with conditional inclusion (SQLite doesn't directly support omit-key, but you can build the JSON via `json_patch` or simply use `CASE WHEN ?N IS NULL THEN NULL ELSE json_object(...) END` and merge). Easier: leave Phase B as-is for omit-NULL (the `null` values are semantically equivalent on read for any sensible consumer; `json_extract(...)` returns NULL for both missing and null keys) and explicitly document this in §4.5 as "consumers MUST treat missing key === null key for backfilled provenance attributes".

**Disposition**: Defer to **T27 verification report decision** — the right time to pin the canonical attributes shape is when the verifier code is being written, because the parity check IS the consumer that cares.

File a note in §4.5 of design.md: "The canonical attributes shape for `edge_kind='provenance', predicate='derived_from'` is **not yet fixed**; Phase B and Phase C currently diverge on `synthesis_timestamp` (C only) and key-omission semantics for NULL legacy columns (C omits, B emits `null`). T27 verifier MUST treat missing key ≡ null key for these attributes, or pin one canonical shape and align both writers."

### FINDING-2 — Important — N+1 FK guard round-trips (T24-r1 FINDING-2 pattern recurs)
**Severity**: Important
**Location**: `crates/engramai/src/substrate/backfill.rs:2255-2267` (per-row FK guard SELECT inside merge loop)

**What's wrong**:
For each of the ~903 production legacy rows, the driver runs:

```rust
let endpoints_ok: bool = tx.query_row(
    "SELECT EXISTS(SELECT 1 FROM nodes WHERE id = ?) \
     AND EXISTS(SELECT 1 FROM nodes WHERE id = ?)",
    params![insight_id, source_id],
    |r: &rusqlite::Row<'_>| r.get::<_, bool>(0),
)?;
```

— one extra database round-trip per row. At production scale: 903 extra round-trips on top of the GROUP BY scan. Smaller than T24's 43k but the pattern is the same. It's foldable into the outer SELECT via two LEFT JOINs to `nodes`:

```sql
LEFT JOIN nodes ni ON ni.id = sp.insight_id
LEFT JOIN nodes ns ON ns.id = sp.source_id
...
-- and add `ni.id IS NOT NULL AND ns.id IS NOT NULL AS endpoints_ok`
-- to the SELECT list
```

The implementer caught the same shortcut in T24 (which fixed it with FINDING-2 in the r1 review). T25 was written immediately after the fix, but the pattern didn't propagate.

**Why it matters**:
- 903 round-trips × ~20µs ≈ 18ms wall-clock. Real but small.
- More importantly: **this is the second occurrence of the same pattern in the same family of drivers within 24h.** That's a signal the FK-guard-via-EXISTS pattern is a default-bad recipe. Future Phase D backfill (e.g., triple extraction T26) will copy this code unless we fix it now.
- Folding into the SELECT is a 4-line edit. Cheap.
- Also: when folded, the `existing_kind` defense-in-depth query (line 2298, same cost issue, T24-r1 FINDING-6) becomes the only remaining per-row round-trip. Worth folding both at once for symmetry.

**Suggested fix**:
1. Extend the outer SELECT to LEFT JOIN nodes twice and select `endpoints_ok` as a derived column.
2. Drop the per-row `endpoints_ok` query.
3. Optionally fold `existing_kind` lookup too: `LEFT JOIN edges e ON e.id = sp.id` selecting `(e.edge_kind, e.predicate) AS existing_kind`.
4. Tests stay green — semantics unchanged.

**Disposition**: Apply now. Same code, cheaper. Propagates pattern to T26.

### FINDING-3 — Minor — Implicit weight/activation parity via column defaults
**Severity**: Minor
**Location**:
  - `crates/engramai/src/storage.rs:1947-1957` (`insert_provenance_edge_row` does NOT set weight or activation)
  - Schema `edges.weight REAL NOT NULL DEFAULT 1.0`, `edges.activation REAL NOT NULL DEFAULT 0.0`
  - Phase B T16 explicitly sets `weight=1.0, activation=0.0`

**What's wrong**:
The T23 helper used by T25 (`insert_provenance_edge_row`) doesn't include `weight` or `activation` in its INSERT, so both fall to the column defaults (`1.0` and `0.0`). Phase B T16 explicitly sets both to those same values. They match TODAY.

But this is **load-bearing implicit parity**. If anyone changes the schema default (a perfectly reasonable migration: "let's default weight to 0.5 for provenance edges so they don't dominate retrieval scoring") — T25's output silently drifts away from Phase B's. The schema change would not appear to touch T25 or Phase B; it would just propagate silently.

**Why it matters**:
- Bugs of this class are hard to detect because tests pass (column default = 1.0, Phase B writes 1.0, they match).
- T22/T23/T24 helpers all have the same shape — they accept a confidence param but not weight/activation, relying on defaults. So this is actually a Phase-C-helper-family issue, not a T25 issue.
- The fix is small: either make the helpers take `weight` and `activation` params explicitly, OR document at the top of `backfill.rs` (with a code-level constant) the "weight/activation come from schema defaults; if schema changes, audit helpers" assumption.

**Suggested fix**:
Add a module-level doc comment in `storage.rs` near `insert_provenance_edge_row` / `insert_structural_edge_row` / `insert_associative_edge_row` explaining the implicit-defaults parity contract, with an explicit pointer to Phase B T16's hardcoded values that they MUST match. No code change required if we accept this is "documented but implicit".

**Disposition**: Defer to ISS-112 batch. 5-line doc comment.

### FINDING-5 — Minor — Counter invariant counts dangling under "skipped_existing" bucket
**Severity**: Minor
**Location**:
  - `crates/engramai/src/substrate/backfill.rs:2390-2403` (`BackfillRun` construction)
  - `crates/engramai/tests/v04_phase_c_backfill_synthesis_provenance.rs::t25_counter_invariant_holds_with_mixed_outcomes`

**What's wrong**:
The struct `BackfillRun` only has three outcome buckets: `rows_inserted`, `rows_skipped_existing`, `rows_failed`. The driver collapses **dangling endpoint + already-existing edge + mismatched-kind** into the single `rows_skipped_existing` bucket. The test `t25_counter_invariant_holds_with_mixed_outcomes` explicitly asserts this collapse: a dangling row goes into `rows_skipped_existing`, not its own bucket.

The audit ROW in `backfill_runs` records the dangling count separately in `notes.rows_skipped_dangling_endpoint`. So an operator inspecting the SQL table sees the breakdown; an operator inspecting the returned struct does NOT.

**Why it matters**:
- Programmatic callers (e.g., a "run all Phase C backfills" orchestrator) get less info than SQL inspection.
- T19's recovery model depends on knowing how many rows were dangling so you can decide whether to re-run T25 after T19.

**Suggested fix**: Either:
- (a) Add a `notes: serde_json::Value` field to `BackfillRun` that mirrors the audit row's notes column. Programmatic callers get full breakdown.
- (b) Add explicit `rows_skipped_dangling: u64` and `rows_skipped_mismatched_kind: u64` fields. More API surface but explicit.
- (c) Accept the API gap; document that fine-grained counts live in `notes`.

The family of drivers ALL have this gap (T22, T23, T24, T25). So (a) or (b) is a cross-cutting refactor.

**Disposition**: Defer to ISS-112 batch. Cross-cutting refactor; pick a direction with a clear migration story for downstream callers.

### FINDING-6 — Minor — `synthesis_timestamp` parse-failure fallback is silent
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:2272-2280`

**What's wrong**:
On RFC3339 parse failure, the driver silently falls back to `utc_now_f64()`. There's no audit counter for parse failures, no log line, no notes field. The `gate_scores` malformed-JSON path is HANDLED visibly (preserved as a string in attributes). The synthesis_timestamp parse failure is not — the bad timestamp just disappears into a "now" value.

**Why it matters**:
- Production today is clean (all rows have valid RFC3339). But a corrupt legacy row would silently produce an edge with `recorded_at = backfill_run_time` instead of the original synthesis time. The forensic value of provenance is lost without operator knowing.
- T20's `created_at` parse-failure path increments `rows_failed_parse` (T20:559). T25 doesn't.

**Suggested fix**: Mirror T20's pattern — add a `rows_failed_parse_timestamp: u64` counter, increment on parse failure, include in `notes`. Tiny change.

**Disposition**: Defer to ISS-112 batch. 5-line edit.
