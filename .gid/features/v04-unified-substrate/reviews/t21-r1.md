## Findings

### FINDING-1 — Critical
**Severity**: Critical
**Location**: `crates/engramai/src/substrate/backfill.rs:766-779` (Pass 1 attribute construction)

**What's wrong**:
The driver layers `entities.metadata` keys on top of the synthetic `{"entity_type": <column>}` using `projected_attrs.insert(k, v)` — which is **last-write-wins**. If `entities.metadata` contains an `entity_type` key, the metadata value overwrites the column value. The design contract in §5.3 says the OPPOSITE:

> `entities.entity_type → nodes.attributes.entity_type`; `entities.metadata`: parse as JSON and **merge keys** into `nodes.attributes` (do not overwrite existing keys; on collision prefer existing).

"Existing" at the merge step = the keys already projected = `entity_type` from the column. So per design, **column wins, metadata loses on collision**.

**Why it matters**:
- Contract violation: T21's behavior diverges from the design doc.
- The test `t21_metadata_entity_type_shadows_column` (lines 88-117 of the test file) ASSERTS the wrong direction, which means the bug is now pinned into the regression suite — any later fix will break the test, but the test is what's wrong, not the code that would satisfy the design.
- Pre-merge `entities` table at production scale (2310 rows) probably has very few rows where `metadata.entity_type` differs from the column — but ANY such row will silently mutate its surface type on backfill.

**Suggested fix**:
1. In Pass 1, change the merge loop from `insert(k, v)` to `entry(k).or_insert(v)` (matches the merge_attributes_existing_wins helper polarity).
2. Rewrite test `t21_metadata_entity_type_shadows_column` to assert: column "person" wins over metadata "organization" (and rename the test accordingly).
3. Update the doc comment at backfill.rs:762-765 — the comment claims shadowing is "intentional (metadata is the more recently edited surface)", which is incorrect reasoning. Replace with the design-§5.3 citation.

### FINDING-2 — Important
**Severity**: Important
**Location**: `crates/engramai/src/graph/store.rs:954-967` (T13 ON CONFLICT clause) interacting with T21

**What's wrong**:
T13's `dual_write_entity_to_nodes` uses `ON CONFLICT(id) DO UPDATE SET attributes = excluded.attributes` — it **clobbers** the entire attributes JSON on every dual-write. T21's Pass 2 protects against the `T13-first → T21-later` ordering (existing-wins merge), but there is **no protection** for the opposite ordering: `T21-first → T13-later`.

Scenario: backfill T21 runs during a migration window. It writes `attributes = {"entity_type": "person", "alias": "al", "note": "founder"}` (the merged legacy view). Then the resolution pipeline processes a memory mentioning the same entity and calls T13 dual-write. T13 overwrites the row's attributes with whatever the in-memory `Entity` struct serializes to — almost certainly LOSING the `alias` and `note` keys that were only in the legacy `entities.metadata`.

**Why it matters**:
- Backfill is a one-shot run; we cannot detect after-the-fact that T13 silently overwrote it.
- The whole point of Pass 2's existing-wins merge is to converge the two write paths to a consistent attributes set. That contract is one-directional under the current T13 SQL.
- Concretely: on the LoCoMo eval, if `entities.metadata` had useful keys (e.g., from a prior knowledge_compile run), they vanish the moment the resolution pipeline re-resolves the entity.

**Suggested fix**:
- Change T13's ON CONFLICT for `attributes` from `excluded.attributes` to a JSON merge with existing-wins semantics. The cleanest path: replace the unconditional set with a function-based merge (custom rusqlite SQL function, or compute the merge in Rust before the UPSERT).
- Track separately as a Phase B follow-up — this is a T13 issue surfaced by T21, not strictly a T21 bug to fix in this commit.
- Document the ordering caveat in T21's module docs until the T13 fix lands.

### FINDING-3 — Important
**Severity**: Important
**Location**: `crates/engramai/tests/v04_phase_c_backfill_entities.rs` (test file as a whole)

**What's wrong**:
Missing tests for these contract-relevant cases:

1. **Re-seeding with mutated metadata** — the only idempotency test (`t21_idempotent_rerun_keeps_attributes_stable`) re-runs the driver with the SAME data. There's no test for: run T21, then update `entities.metadata` for the same id, then re-run T21 — what does Pass 2's existing-wins do with the new metadata keys? Expected: new keys land, existing keys stay. Should be tested explicitly.

2. **Metadata containing reserved column keys** — what if `entities.metadata = '{"namespace": "other", "id": "different"}'`? Both Pass 1 and Pass 2 currently merge these into `attributes` as plain keys — they CAN'T corrupt the column values (column is separate). But there's no test pinning that behavior, so a future "flatten attributes into columns" optimization would silently corrupt namespacing.

3. **`entity_type` empty string** — `entities.entity_type` is NOT NULL but could be `''`. No test for that case.

4. **Pass 2 with corrupt existing attributes** — if a `nodes` row has `attributes='null'` or `attributes='[]'` (corrupt data from a prior bug), `merge_attributes_existing_wins` silently returns the original string and drops the legacy metadata. No telemetry surfaces this. No test pins the behavior.

**Why it matters**:
Each gap is a contract clause not pinned by a test. Future refactors can break them silently.

**Suggested fix**:
Add four targeted tests. Don't bundle them with FINDING-1's fix — that's a separate commit. Tests 1-3 are pure additions; test 4 should also bump a counter in the audit row (`rows_existing_attrs_not_object`) so production occurrences are visible.

### FINDING-4 — Important
**Severity**: Important
**Location**: `crates/engramai/src/substrate/backfill.rs:705-715` (audit row open) and 826-841 (audit row close); same pattern in T19/T20

**What's wrong**:
The `backfill_runs` audit row is INSERTed BEFORE the tx (outside, with `finished_at=NULL`), then UPDATEd AFTER `tx.commit()`. Three failure windows are unprotected:

1. **Crash during tx** → tx rolls back, audit row remains `finished_at=NULL, rows_*=0` forever. Operator has to manually distinguish "in-progress" from "crashed" runs by wall-clock heuristic.
2. **Crash between commit and UPDATE** → data is committed but audit row shows zero rows. Misleading metrics.
3. **Concurrent runs** (same legacy_table, different namespaces) → two open audit rows; impossible to tell from the audit table alone which run owned which inserts.

Same pattern in T19/T20, so this is a cross-cutting issue, not T21-specific.

**Why it matters**:
- The audit table is the only operator-visible signal for "did backfill complete?" If it lies, backfill becomes unobservable.
- Real-world: a partial backfill run leaves the DB in an inconsistent state (some rows projected, some not) but the operator can't tell because the audit row says "0 inserts".

**Suggested fix**:
Pick one:
- (A) Make the audit INSERT and UPDATE both run INSIDE the same tx as the work. Crash before commit → no audit row at all (consistent with no data). This is the cleanest.
- (B) Keep current shape but add a startup-recovery step that scans `backfill_runs WHERE finished_at IS NULL AND started_at < now() - 1h` and marks them as `failed_or_orphaned`.

Recommendation: (A). Refactor the audit-row helper to take `&Transaction` and call it from inside `tx`. Apply to T19, T20, T21 in one cleanup commit.

### FINDING-5 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:822-825` (Pass 2 unconditional UPDATE)

**What's wrong**:
Pass 2 always executes `UPDATE nodes SET attributes = ?, updated_at = ? WHERE id = ?` even when the merge result is byte-identical to the existing attributes. This bumps `updated_at` on every rerun.

**Why it matters**:
- Downstream consumers using `updated_at > X` as a "changed since X" filter will see false positives after every backfill rerun.
- Test `t21_idempotent_rerun_keeps_attributes_stable` only checks attributes byte-equality, NOT `updated_at` stability — so the noise is invisible to the test suite.

**Suggested fix**:
Diff the merge result against the existing attributes; skip the UPDATE if they're identical. Or: accept the noise as documented behavior and add an explicit comment in the docstring saying "Pass 2 always bumps updated_at; consumers MUST use `attributes` content equality, not `updated_at`, to detect change".

### FINDING-6 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:835-840` (audit notes JSON)

**What's wrong**:
`rows_kind_mismatch` is a STRICT SUBSET of `rows_skipped_existing` (every kind-mismatch was first counted as skipped). Audit JSON exposes both side-by-side with no documentation that one is a subset of the other.

Concretely: `rows_read=10, rows_inserted=5, rows_skipped_existing=5, rows_kind_mismatch=2` means "5 skipped total, of which 2 were because the existing row was a different node_kind". An audit reader summing `inserted + skipped + kind_mismatch` would get 12 instead of 10 and conclude the invariant is broken.

**Why it matters**:
Internal-only telemetry, but a dashboard or alerting rule could silently double-count.

**Suggested fix**:
Add a comment to the notes JSON construction explicitly stating the sub-category relationship. Better: rename to `rows_skipped_kind_mismatch` so the prefix `rows_skipped_*` signals "subset of rows_skipped_existing".

### FINDING-7 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:651-657` (merge_attributes_existing_wins) — non-Object existing

**What's wrong**:
If the existing `nodes.attributes` value is valid JSON but not an object (`null`, `[]`, `"string"`, etc.), `merge_attributes_existing_wins` silently returns the existing value unchanged and DROPS the new keys. No counter, no log, no audit signal.

**Why it matters**:
If a future bug or manual edit leaves a `nodes` row with `attributes='null'`, T21 Pass 2 silently fails to migrate the legacy metadata. The operator only finds out when a downstream query returns missing data.

**Suggested fix**:
Surface this in the BackfillRun audit notes as `rows_existing_attrs_not_object`. Don't fail the row — same defensive philosophy as malformed metadata. Combine with FINDING-3 test #4.
