### FINDING-1 — Important
**Severity**: Important
**Location**:
  - `crates/engramai/src/substrate/backfill.rs:1660-1672` (driver hash template + justifying comment)
  - `.gid/features/v04-unified-substrate/design.md` §5.3 hebbian_links amended template (the table this is reviewing AGAINST)

**What's wrong**:
The amended §5.3 template (committed two commits ago in `ae0ca57` as part of T23-r1 fixes) explicitly says:

```
hash_input = "hebbian_links|<memory_id>|<related_id>|<namespace>|<edge_kind>|<predicate>"
```

— SIX pipe-delimited tokens, no signal_source. The driver uses SEVEN tokens (adds signal_source after namespace). The module comment at line 1635-1650 justifies the deviation: §4.3 declares signal_source a row-identity dimension via the partial unique index `idx_edges_assoc_unique`, so future multi-signal production data would collide on the primary id and be silently rejected by `INSERT OR IGNORE` before the partial unique index could catch it. The justification is correct on the merits — but it is exactly the class of **code-vs-doc drift** T23-r1 FINDING-1 flagged: §5.3 is supposed to be the single source of truth for hash layouts so future backfill implementers can copy-paste without re-deriving the contract. Now T25 implementer (next, for synthesis_provenance) has to read both §5.3 AND the t24 module comment to know what to do.

This is the second time in two backfills the §5.3 template has been wrong. The pattern is clearer now: §5.3 was authored before §4.3's row-identity-dimension contract was fully internalized, and §4.3 is the binding constraint for hebbian_links.

**Why it matters**:
- Doc-vs-code drift compounds. T22, T23, T24 each have a minor twist relative to §5.3. By T25 there will be ~3 "consult the module comment for the real story" cases.
- §4.3's row-identity contract is BINDING per the partial unique index it built. §5.3 ignoring it is the bug.
- T25 (synthesis_provenance) is up next. Its design §5.3 template currently reads `synthesis_provenance|<id>|<source_id>|<edge_kind>|<predicate>` — five tokens. Is that complete? Per §4.3 + §3.2 there's no analogous partial unique index for `provenance` edges. So T25's template is actually correct as-is. But the T25 implementer can't know that without reading both sections.

**Suggested fix**:
Edit `design.md` §5.3 hebbian_links template to read:

```
# hebbian_links (composite natural key + signal_source as §4.3
# row-identity dimension):
hash_input = "hebbian_links|<memory_id>|<related_id>|<namespace>|<signal_source>|<edge_kind>|<predicate>"
```

with a one-line note: "signal_source is included because §4.3 makes it a row-identity dimension via `idx_edges_assoc_unique`; a future write of the same canonical pair with a different signal_source MUST get a distinct edge."

Tiny edit. Removes the §5.3-vs-driver-comment second-source-of-truth.

**Disposition**: Apply now (before T25 implementer reads §5.3).

### FINDING-2 — Important
**Severity**: Important
**Location**: `crates/engramai/src/substrate/backfill.rs:1876-1886` (per-group `SELECT COUNT(*)` inside the merge loop)

**What's wrong**:
For each merged group the driver re-queries:

```rust
let legacy_count: i64 = tx.query_row(
    "SELECT COUNT(*) FROM hebbian_links \
     WHERE namespace = ? \
       AND COALESCE(signal_source, 'corecall') = ? \
       AND ((source_id = ? AND target_id = ?) OR (source_id = ? AND target_id = ?))",
    params![ns_val, sig_src, lo, hi, hi, lo],
    |r| r.get(0),
)?;
```

— ONE extra database round-trip per group. At production scale (43,346 rows producing 43,227 distinct canonical-pair groups), that is **43,227 extra round-trips** on top of the GROUP BY scan. Each round-trip is small but the cumulative cost is real even for a one-shot backfill, and the alternative is trivial: add `COUNT(*) AS legacy_count` to the existing GROUP BY SELECT, returning it as one more column.

The implementer's own comment at line 1872-1875 acknowledges this:

> Cost: 1 small query per merged group. Total ~43k production groups → 43k extra round-trips; acceptable for a one-shot backfill. If it ever needs optimizing, fold COUNT(*) into the SELECT above.

So the implementer KNEW the fix and deferred it. That's a self-aware shortcut, but T25 (synthesis_provenance) is the last structural Phase C task and copies this pattern. If the shortcut propagates, then verification report (T27) and any operator dashboard built on backfill_runs will have the same anti-pattern.

**Why it matters**:
- 43k extra round-trips at SQLite latency (~20µs each on M-class SSD) ≈ 0.9 s wall-clock penalty for production backfill. Not catastrophic, but unnecessary.
- Sets the pattern for T25. Fixing now means the next ~3 backfill drivers inherit the clean pattern by copy.
- Asymmetry: every other Phase C driver has `rows_inserted = COUNT(legacy rows)`. T24 is the only one with SQL-side merge, and the natural place to expose the row count is INSIDE the GROUP BY result.

**Suggested fix**:
1. Add `COUNT(*) AS legacy_count` to the outer GROUP BY SELECT (one extra column).
2. Read it into a new field of `HebRow`.
3. Delete the inner `SELECT COUNT(*)` per-group query.
4. Update the comment block.

Net diff is ~15 lines smaller, ~30k production round-trips saved.

**Disposition**: Apply now. Cheap, propagates to T25, removes self-acknowledged debt.

### FINDING-3 — Important
**Severity**: Important
**Location**:
  - `crates/engramai/src/storage.rs:2092-2129` (`insert_associative_edge_row` helper, `confidence = 1.0` hardcoded in SQL)
  - Compare against T22/T23 helpers also using `confidence = 1.0`

**What's wrong**:
The helper hardcodes `confidence = 1.0` for every associative edge it inserts. Legacy `hebbian_links` has no confidence column, so the implementer chose `1.0` as a sensible default. T22's `insert_structural_edge_row` and T23's `insert_provenance_edge_row` both use `confidence = caller-supplied`, but **every** Phase C call site passes `1.0`. T14 Phase B dual-write also writes `confidence = 1.0` (see graph/store.rs:1198 in the original Phase B helper).

So the de facto policy is: "anything backfilled from authoritative legacy data is `confidence = 1.0`". But this is implicit, never written down. §3.2 schema says `edges.confidence DEFAULT 0.5`; §4.3 doesn't define associative-edge confidence semantics; §5.3 doesn't mention confidence in the field mapping for any link table.

The risk: a future writer (e.g., a Phase D LLM-derived associative edge from triple extraction) might write `confidence = 0.7` or `confidence = inferred-score`. Then a reader of `edges` doing `WHERE confidence > 0.8` would include all backfilled associative edges (legacy-authoritative) AND exclude all LLM-derived ones — which **could** be the desired semantics OR could be exactly inverted from the operator's mental model. There's no doc to consult.

**Why it matters**:
- Implicit policy is fragile. Whichever way Phase D readers happen to interpret `confidence > X` first becomes load-bearing.
- T25 (synthesis_provenance) HAS a confidence column in legacy. The implementer will plausibly pass it through. Then synthesis_provenance edges have varied confidence values (0.0–1.0) while every other Phase C edge has 1.0. A reader can no longer rely on "1.0 = backfilled" or any uniform interpretation.
- Trivial to document, ugly to retrofit later.

**Suggested fix**:
Add a paragraph to design.md §5.3 (or §4.3 for associative-specific):

> **Confidence semantics in backfilled edges.** Phase C backfill drivers assign `edges.confidence = 1.0` to all edges projected from legacy tables that lack a confidence column (entity_relations, memory_entities, hebbian_links — i.e. T22, T23, T24). Rationale: legacy data is the system's authoritative observation; downstream readers should treat backfilled edges as ground-truth. Drivers projecting from tables WITH a confidence column (synthesis_provenance / T25) pass the legacy column through verbatim. Phase D writers (LLM-derived associations, gate-scored synthesis, etc.) MAY emit confidence < 1.0 to reflect uncertainty.

Then add a top-of-`backfill.rs` constant `const BACKFILL_LEGACY_CONFIDENCE: f64 = 1.0;` referenced by the helpers, so the policy has ONE place to change.

**Disposition**: Defer to T25 (sensible to make this decision once T25's confidence pass-through is concrete). File as ISS-112 batch.

### FINDING-4 — Minor
**Severity**: Minor
**Location**: `crates/engramai/tests/v04_phase_c_backfill_hebbian.rs` (test file overall)

**What's wrong**:
The collision test (`t24_collision_pair_merges_with_sum_semantics`) exercises 2 legacy rows merging. No test exercises 3+ rows in a single group. Production today can't produce 3-way collisions because legacy PK `(source_id, target_id)` blocks duplicate-direction rows — so at most you get one `(A,B)` row + one `(B,A)` row = 2-way.

But the SQL `GROUP BY (canonical_lo, canonical_hi, namespace, signal_source)` merge is N-ary. If a future migration relaxes the legacy PK (e.g., adds signal_source to it), 3-way merges become possible. The SUM-semantics IS correct N-ary, but it's not pinned by a test.

**Why it matters**:
- Low likelihood (legacy PK is stable). High blast radius if it ever fires.
- A 3-way test is trivial: seed two `(A,B)` rows with different signal_source + one `(B,A)`, run with namespace filter that captures all, assert merge counts.
  
  Wait — actually the legacy PK blocks that. The 3-way scenario requires the SQL merge to see >2 rows in one group, which requires bypassing the PK. The only way to construct it in a test is to seed via raw SQL with a different PK or set up the legacy schema without the PK first. That's noisy.

**Suggested fix**: Skip a direct N>2 test (artificial); instead add a comment in the driver near the merge loop documenting that the SQL is N-ary safe even though current legacy PK bounds collisions to 2-way, and link to this finding.

**Disposition**: Defer to ISS-112 batch as a 2-line comment.

### FINDING-5 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:1932-1945` (CSV-split parsing of `GROUP_CONCAT(DISTINCT ...)` results)

**What's wrong**:
The driver consumes `GROUP_CONCAT(DISTINCT direction)` and `GROUP_CONCAT(DISTINCT COALESCE(signal_detail, ''))` by splitting on `','`:

```rust
let directions: Vec<String> = dirs_csv.split(',').filter(...).collect();
```

This is fragile. If any legacy `direction` or `signal_detail` value ever contains a literal comma, the split parses it as two distinct values. Production today is safe — `direction` is always `'bidirectional'` (no comma) and `signal_detail` is always empty. But:

- `direction` is `TEXT NOT NULL DEFAULT 'bidirectional'` with no CHECK constraint. A migration or operator could write `"directed,reverse"` and break the parser.
- `signal_detail` is documented in Phase B `dual_write_hebbian_to_edges` (graph/store.rs:1146) as "opaque blob, callers may pass JSON like `{\"entity_overlap\":0.4}`". JSON contains commas. **However**: Phase B's helper writes signal_detail ONLY to `edges`, not to legacy `hebbian_links`. The only writers to legacy `hebbian_links.signal_detail` are direct SQL paths (none currently), so the comma-in-JSON risk is bounded by "no current writer". Future-but-not-imminent.

**Why it matters**:
- Production-safe today, future-fragile.
- The fix is small.

**Suggested fix**:
Either:
- (a) Replace `GROUP_CONCAT(DISTINCT direction)` with a sub-SELECT that materializes a Rust-side `HashSet<String>`. Costs one more query per group (and FINDING-2 already says to eliminate per-group queries — incompatible).
- (b) Use SQLite's `GROUP_CONCAT(DISTINCT direction, x'1f')` with ASCII unit-separator (chr 31) as the delimiter. Then `.split('\u{1f}')` is collision-free for any printable text.
- (c) Just document the assumption inline ("legacy direction/signal_detail values MUST NOT contain commas") and rely on the lack-of-writers safety today.

Recommend (b) — defense-in-depth, two-character change.

**Disposition**: Defer to ISS-112. Note in the inline comment near the parse.

### FINDING-6 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:1960-1991` (insert + audit-bucket increment block)

**What's wrong (or not)**: The `rows_skipped_mismatched_kind` defense-in-depth bucket. Under correct contract it never fires because:
- The hash input is prefixed with `"hebbian_links|"` so a T24 id never collides with a T22 (`"entity_relations|"`) or T23 (`"memory_entities|"`) id.
- The hash input encodes canonical pair + namespace + signal_source uniquely, so two T24 calls can never produce the same id with different intent.

So the `existing_kind` SELECT is paranoid. Same pattern landed in T22 and T23 — consistent across the family.

The cost is one extra round-trip per non-dangling group. That's 43,227 - 0 = 43,227 extra round-trips in production. Combined with FINDING-2's COUNT(*) round-trips, the driver does ~86k extra trips it could skip.

**Why it matters**: The defense is genuinely useful — if a future contributor changes the hash template and forgets to re-derive ids, this bucket surfaces the corruption immediately instead of producing wrong edges silently. But the cost-benefit ratio in steady-state is poor.

**Suggested fix**: Keep the defense for the FIRST run of a fresh DB (when the bucket might catch a real bug). Skip it on repeated runs by setting a flag once we've inserted enough rows to know the hash template is correct. Or just accept the cost — 43k round-trips × 20µs ≈ 0.9 s, real but small.

Actually the cleanest fix: drop the defense entirely and replace it with a single post-insert verification query:

```rust
let mismatched: i64 = tx.query_row(
    "SELECT COUNT(*) FROM edges WHERE id IN (...inserted ids...) \
     AND (edge_kind != 'associative' OR predicate != 'co_activated')",
    ...
)?;
```

— but at that point you're materializing N inserted ids, which is its own cost. Probably not worth it.

**Disposition**: Accept the cost. No code change.

### FINDING-7 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:1983-1992` (audit-bucket mutual exclusion)

**What's wrong (or not)**: The audit-counter increments are correctly mutually exclusive per loop iteration:

1. If `endpoints_ok == false` → `rows_skipped_dangling_endpoint += legacy_count`, `continue`. (FK check at line 1898-1910.)
2. Otherwise `existing_kind` is queried.
3. Then `insert_associative_edge_row` runs.
4. If `inserted == true` → `rows_inserted += legacy_count`. (Line 1984.)
5. Else if `existing_kind == Some((kind, pred))` with mismatched kind/pred → `rows_skipped_mismatched_kind += legacy_count`. (Line 1986-1988.)
6. Else if `existing_kind == Some(...)` with matched kind/pred → `rows_skipped_existing += legacy_count`. (Line 1988-1990.)
7. Else (existing_kind is None but `inserted == false`) → `rows_skipped_existing += legacy_count`. (Line 1991.)

Case 7 is interesting — can it happen? `inserted == false` means `INSERT OR IGNORE` rolled the row, which means a row with the same id existed. So `existing_kind` MUST be `Some(...)`. Case 7 is dead code.

**Why it matters**: Defense-in-depth — if a race or sibling tx inserts and deletes between the existing_kind SELECT and our INSERT, the lookup could miss. In a single-tx context (which this is — line 1864 `tx.commit()`) this is impossible. So case 7 truly is unreachable.

**Suggested fix**: Replace case 7 with `unreachable!("insert_or_ignore rejected but existing_kind None; impossible in single-tx context")`. Surfaces any future regression that breaks the tx boundary.

**Disposition**: Defer to ISS-112 batch. 1-line code change.
