### FINDING-1 — Important
**Severity**: Important
**Location**: `crates/engramai/src/substrate/backfill.rs:1295-1315` (driver docstring) and design.md §5.3 lines 1140-1148

**What's wrong**:
The implementation follows design §3.3 (the normative kind/predicate table that splits memory_entities by role: `mention/''/triple/unknown → provenance/mentions`; `subject → structural/subject_of`; `object → structural/object_of`). But design §5.3 prose at line 1140 still says:

> `memory_entities → edges (kind=provenance, predicate=mentions)` (9237 rows). **Field mapping**: `memory_entities.role → edges.predicate` (role='mention' → predicate='mentions', role='subject' → predicate='subject_of', etc.).

This collapses everything into `kind=provenance` and is **wrong** — a `subject` role should produce `kind=structural` per §3.3 line 320. The commit message flagged this but deferred the fix to "a follow-up doc commit". The deferral is real tech debt: anyone reading §5.3 in isolation (which is what backfill implementers will do for T24/T25) will infer the wrong kind for subject/object roles. T22's `entity_relations` driver is one section over and could easily inherit the same mistake.

**Why it matters**:
- Design-doc internal contradiction (§3.3 normative ≠ §5.3 prose) survives in tree.
- T24 (hebbian_links) is up next and §5.3 line 1147 says `kind=associative, predicate=co_activated`. If the §5.3 prose has already been shown to be loose (this finding proves it), the T24 implementer can't trust §5.3 alone; they have to cross-check §3.3 every time. That's friction the design doc is supposed to eliminate.
- "Follow-up doc commit" is the classic patch-not-root-fix pattern — exactly what SOUL.md says to avoid.

**Suggested fix**:
Edit `.gid/features/v04-unified-substrate/design.md` §5.3 line 1140 to read:

```
`memory_entities → edges` (9237 rows), split by role per §3.3:
  - `role='mention'`/`''`/`'triple'`/unknown → `kind='provenance', predicate='mentions'`
  - `role='subject'` → `kind='structural', predicate='subject_of'`
  - `role='object'`  → `kind='structural', predicate='object_of'`
```

Tiny edit. Should be amended into the T23 commit family before T24 starts, so T24 implementer sees a consistent §5.3.

**Disposition**: Apply now (before T24).

### FINDING-2 — Important
**Severity**: Important
**Location**: `crates/engramai/src/substrate/backfill.rs:1402` (hash input layout) and design.md §5.3 lines 1170-1186

**What's wrong**:
The design contract for non-PK source tables says:

```
edges.id = uuid_from_hash(sha256(
    source_table || '|' ||
    source_id    || '|' ||      -- the row's primary key in legacy table
    target_id    || '|' ||
    edge_kind    || '|' ||
    predicate
))
```

with the caveat "for tables that lack a single PK column, use the smallest UNIQUE tuple as `source_id` (e.g., `memory_entities` uses `(memory_id, entity_id, role)`)".

The driver inlines all three components flat:

```
"memory_entities|{memory_id}|{entity_id}|{role}|{edge_kind}|{predicate}"
```

— 6 pipe-delimited tokens, not the 5 the design formula declares. The design example for memory_entities literally lists `(memory_id, entity_id, role)` as the natural source_id tuple, which would parse as either:
  (a) `source_id = "memory_id|entity_id|role"` and `target_id` = ???, or
  (b) what I did: flatten all components into the canonical layout.

Reading (a) is incoherent because `memory_entities` IS a link table — `memory_id` is the natural source and `entity_id` is the natural target. So reading (b) is the only internally consistent interpretation. **No bug.** But the design text is ambiguous enough that a reviewer (or T24 implementer doing the same hash for hebbian_links) could pick reading (a) and produce non-interoperable ids.

**Why it matters**:
- Determinism is the whole point of the hash; two different drivers using two different interpretations of the same spec would produce different ids for "the same" row, breaking idempotency across crate versions.
- T24 has the same problem: hebbian_links' natural key is `(memory_id, related_id, namespace)` per design §5.3 line 1185 — flat or nested?

**Suggested fix**:
Edit design.md §5.3 to disambiguate. Concrete example:

```
For link tables (memory_entities, hebbian_links), inline all
discriminators flat in the canonical natural-key order:

  hash_input = "memory_entities|<memory_id>|<entity_id>|<role>|<edge_kind>|<predicate>"
  hash_input = "hebbian_links|<memory_id>|<related_id>|<namespace>|<edge_kind>|<predicate>"

The number of pipe-delimited tokens is fixed per source table.
```

**Disposition**: Apply now (before T24, same window as FINDING-1). Saves T24 implementer from re-deriving the convention.

### FINDING-3 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:1413-1421` (FK pre-check query)

**What's wrong (or not)**: The FK guard query is:

```sql
SELECT EXISTS(SELECT 1 FROM nodes WHERE id = ?) AND
       EXISTS(SELECT 1 FROM nodes WHERE id = ?)
```

with `params![memory_id, entity_id]`. This relies on rusqlite's anonymous-positional `?` binding being **sequential** (each `?` consumes the next param). I confirmed via rusqlite docs that this is correct — `?` is anonymous positional, not reused. If a future contributor wonders whether both `?` accidentally point at `memory_id`, the answer is no.

**Why it matters**: Nothing's broken. But the query reads ambiguously if you don't know rusqlite's binding semantics. A reader who's used to PostgreSQL's `$1`/`$2` numbered binding might worry.

**Suggested fix**: Either switch to numbered binding for clarity:

```rust
"SELECT EXISTS(SELECT 1 FROM nodes WHERE id = ?1) AND
        EXISTS(SELECT 1 FROM nodes WHERE id = ?2)"
```

or add a single-line comment `// rusqlite uses sequential anonymous binding; ?#1 = memory_id, ?#2 = entity_id`.

**Disposition**: Skip. Pattern matches existing rusqlite code in the crate; flipping just one site to numbered is inconsistent. Defer to a separate sweep if anyone ever introduces a `?n` style elsewhere.

### FINDING-4 — Minor
**Severity**: Minor
**Location**: `crates/engramai/tests/v04_phase_c_backfill_memory_entities.rs` (test file overall)

**What's wrong**: No test exercises cross-namespace memory→entity link. Every test has memory and entity in the same NS. Concrete missing scenario: memory `mem-1` in `ns-a` has a `memory_entities` row pointing at entity `ent-x` in `ns-b`. The driver writes `edges.namespace = ns-a` (derived from parent memory). This is correct per design intent (the edge belongs to the relationship's observation NS, not the endpoint's NS), but it's not asserted anywhere.

**Why it matters**:
- Production engram has cross-NS entity sharing (the v0.3 graph layer explicitly supports it). If T23 silently broke cross-NS links, the bench would degrade and we'd hunt for it later.
- T24 will have the exact same question: hebbian_links has an explicit `namespace` column, so it's slightly different — but the test bar should be the same.

**Suggested fix**: Add one integration test:

```rust
#[test]
fn t23_cross_namespace_link_inherits_memory_namespace() {
    // mem-1 in ns-a, ent-1 in ns-b, link row between them.
    // Assert edges.namespace == 'ns-a' and target_id resolves to
    // the ns-b entity node.
}
```

~25 lines. Adds confidence + asserts the namespace contract explicitly.

**Disposition**: Apply now if cheap (it is), otherwise file as ISS-112 batch.

### FINDING-5 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:1366` (`rows_failed: u64 = 0`, set once, never incremented)

**What's wrong**: The driver hard-codes `rows_failed = 0` because every row either inserts, skips-existing, skips-dangling, or skips-mismatched-kind. There's no parse-failure path (no metadata JSON to parse), no LLM call. Good. But `BackfillRun::assert_counter_invariant` only triggers if `rows_read != inserted + skipped + failed`, and since `failed` is always 0 here the invariant only checks `read == inserted + skipped`. If a future refactor adds a parse path and forgets to increment `rows_failed`, the assert wouldn't catch the drift.

**Why it matters**: Defense-in-depth. The mismatched-kind test (`t23_mismatched_kind_at_same_id_counted_not_silently_skipped`) proves the pattern works for the buckets that ARE used. The `failed` bucket has no proof.

**Suggested fix**: None for T23 (would be inventing failure paths just to test the assert). Note for T24/T25: hebbian_links has a `signal_source` field that could theoretically fail to project; that driver's test suite should exercise the failed-counter path.

**Disposition**: Note for T24, no code change for T23.

### FINDING-6 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/storage.rs:1934-2008` (`insert_provenance_edge_row`) vs `:1845-1894` (`insert_structural_edge_row`)

**What's wrong**: The two helpers are ~95% identical SQL with `edge_kind` being the only literal difference. The commit notes this as a deliberate choice (helper-per-kind for type-level safety and grep-ability). Arguments for the choice:
- Caller can't write `edge_kind='provenance'` with the structural helper by accident — the literal is baked into the SQL.
- Searching for `'provenance'` finds writer call sites directly without false positives in JSON attribute strings.

Arguments against:
- DRY. 5 edge kinds × ~75 lines = ~375 lines of nearly-duplicated SQL once T24 and T25 land.
- Adding a column to `edges` (e.g., a new audit field) requires N edits.

**Why it matters**: Becomes more painful as more kinds land. After T24 (associative) and T25 (provenance/derived_from — same kind as memory_entities/mentions but different predicate), there will be 4 nearly-identical helpers. At that point the unified `insert_typed_edge_row` becomes the right move.

**Suggested fix**: Defer. After T24+T25 land, evaluate consolidating into `Storage::insert_typed_edge_row(tx, kind, predicate, ...)` if the kind-specific differences are zero. The decision is easier to make with the full set of writers in tree.

**Disposition**: Note for post-T25 refactor. File as part of ISS-112 sweep.

### FINDING-7 — Minor
**Severity**: Minor
**Location**: `crates/engramai/src/substrate/backfill.rs:1457-1462` (the unreachable arm)

**What's wrong (or not)**: The driver dispatches on `edge_kind`:

```rust
match edge_kind {
    "structural" => Storage::insert_structural_edge_row(...),
    "provenance" => Storage::insert_provenance_edge_row(...),
    _ => unreachable!(
        "role_to_kind_predicate only emits 'structural' or 'provenance'"
    ),
}
```

This is correct today. But `role_to_kind_predicate` is private to this module and could be edited in a future commit to emit, e.g., `"associative"` for some weird role. The `unreachable!` would then panic in production. A type-level fix would replace `&'static str` with an enum:

```rust
enum BackfillEdgeKind { Structural, Provenance }
fn role_to_kind_predicate(role: &str) -> (BackfillEdgeKind, &'static str, bool) { ... }
```

— makes the match exhaustive at compile time.

**Why it matters**: It's a tiny correctness-by-construction win. Not worth refactoring T23 alone for it, but if T24/T25 ever extend the kind set, the enum starts paying for itself.

**Suggested fix**: Defer. Revisit if any future change extends `role_to_kind_predicate`'s output set.

**Disposition**: Note, no action.
