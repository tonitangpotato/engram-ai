# Requirements: Topic Compilation & Feedback

This feature covers the core compilation pipeline: generating topic pages from engram memories, discovering topics automatically, incremental recompilation, structural operations (merge/split), cross-topic linking, and user feedback mechanisms that let users correct, rate, and preview compilation results. See [master requirements.md](../requirements.md) for GUARDs (system-wide constraints GUARD-1 through GUARD-6).

---

## GOALs — Functional Requirements

### §1 — Topic Page Compilation（编译层）

#### GOAL-comp.1: Topic Page Generation [P0]
The system compiles related memories into a browseable knowledge page that summarizes key insights, attributes each insight to its source memories, and links to related topics. The page must be human-readable and contain enough context to be useful without referring back to source memories.

**Pass/fail**: Given a cluster of ≥3 related memories, the system produces a topic page that: (1) presents a coherent summary of the topic, (2) ensures each key point traces to at least one source memory (GUARD-4 compliance), (3) links to related topics when they exist.

#### GOAL-comp.2: Automatic Topic Discovery [P0]
系统自动从 engram 记忆库发现应该编译成 topic pages 的主题。发现机制复用 synthesis engine 的 cluster discovery（4信号聚类），但阈值和粒度可独立配置。A topic is only created when the cluster contains ≥3 memories (consistent with GOAL-comp.1 compilation minimum). Clusters with fewer memories are tracked but not compiled.

**Embedding reuse**: Topic discovery MUST use engram's existing embeddings from the `memory_embeddings` table (generated by the configured embedding provider, e.g., Ollama nomic-embed-text). KC does not maintain its own embedding pipeline for source memories — it reads what engram already computed. When a memory has no embedding (e.g., embedding provider was offline when the memory was stored), the system falls back to a deterministic hash-based pseudo-embedding with a warning. This fallback produces lower-quality clustering and should be treated as degraded mode, not normal operation.

**Pass/fail**: 对一个有 50+ 条记忆的 engram DB 运行 topic discovery，输出 ≥1 个待编译 topic，每个 topic 有 member IDs、quality score (∈ [0.0, 1.0] representing cluster coherence, where ≥0.7 means ready to compile), suggested title。Discovery must use real embeddings when available in `memory_embeddings`.

#### GOAL-comp.3: Incremental Compilation [P0]
当新记忆加入或现有记忆被 recall（Hebbian 强化）时，系统识别受影响的 topic pages 并标记为 stale。下一次编译周期只重编译 stale pages。Staleness detection occurs at memory insertion time (synchronous) or at the start of each compilation cycle (batch). The choice is a design decision, but detection must complete before compilation begins.

**Pass/fail**: 添加一条新记忆后，系统正确标记包含相关 cluster 的 topic page 为 stale，且不标记无关 pages。

#### GOAL-comp.4: Topic Merging [P1]
当两个 topic pages 的内容重叠度超过可配置阈值时，系统建议合并（或自动合并）。合并后保留两者来源，更新所有引用。Overlap is measured as the Jaccard similarity of source memory sets between two topic pages. Default threshold: 60%. Configurable range: 30%–90%.

**Pass/fail**: 创建两个重叠 >60% 的 topic pages，系统检测到重叠并产出合并建议或执行合并，合并后 page 的 provenance 包含两个原 page 的所有源记忆。

#### GOAL-comp.5: Topic Splitting [P1]
当一个 topic page 增长超过可配置上限（默认 15 个要点）时，系统建议拆分成子 topics。拆分后原 page 变为 index page。Sub-topics should be semantically coherent — each sub-topic's key points should be more similar to each other than to points in other sub-topics. The original page becomes an index linking to sub-topics.

**Pass/fail**: 一个 topic page 有 20 个要点时，系统检测到并产出拆分建议，建议包含子 topic 划分和每个子 topic 的 member IDs。

#### GOAL-comp.6: Cross-Topic Linking [P1]
Topic pages are automatically linked when they share content or concepts, with links categorized by relationship type and weighted by strength. Links are based on: shared entities, shared source memories, and embedding similarity. Cross-topic links are computed during compilation (when a topic page is compiled or recompiled).

**Pass/fail**: When two topic pages share ≥2 source memories, the system automatically creates a link between them, with link strength reflecting the degree of sharing.

#### GOAL-comp.10: Compilation Failure Handling [P0]
If compilation fails for any topic page (LLM API unavailable, unparseable output, database write failure), the system preserves the previous version of that page, logs the failure reason, and marks the page as still-stale for retry. Successfully compiled pages in the same batch are not rolled back. LLM costs for failed compilations are recorded.

**Pass/fail**: When compilation fails for one topic page in a batch of three, the other two pages are successfully compiled and saved, while the failed page remains in stale state with error logged.

### §2 — Compilation Feedback & User Control（编译反馈与用户控制）

#### GOAL-comp.7: Manual Topic Edit [P0]
Users can manually edit topic page content (correcting errors, adjusting wording, removing irrelevant points). User edits are preserved across recompilation — the compiler does not overwrite user modifications. When a page with user edits is recompiled due to new source material, user-edited points are preserved unchanged. New content is added around user edits. If new source material contradicts a user edit, the conflict is flagged (see maintenance/conflict detection) rather than silently resolved.

**Pass/fail**: After a user modifies the wording of a key point, subsequent recompilation preserves the user's version while incorporating any new points from additional source memories.

#### GOAL-comp.8: Point-Level Feedback [P1]
用户可以对 topic page 的单个 key point (要点) 标记：✅ correct、❌ wrong、🔄 outdated。标记为 wrong 的 key point 在下次编译时被移除或重新生成。标记为 outdated 的 key point 触发重新编译。

**Pass/fail**: 用户标记某 key point 为 ❌ wrong，下次编译后该 key point 从 page 中消失或被替换。标记 ✅ correct 的 key points 在后续编译中被优先保留。

#### GOAL-comp.9: Compilation Dry Run [P1]
Users can preview compilation effects (which pages will change, estimated LLM cost) before executing. Users can also skip confirmation for automation purposes. Dry-run shows estimated cost. If estimated cost exceeds a configurable user-budget threshold, compilation requires explicit confirmation even when confirmation is normally skipped.

**Pass/fail**: A dry-run operation displays the list of topic pages that would be compiled or updated, along with estimated token count and cost, without modifying any data. Actual compilation can proceed after user confirmation or with confirmation skipped for automation.
