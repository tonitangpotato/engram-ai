## TL;DR

- **Code reality**: 4 of 5 v0.3 features are substantially implemented
  (graph-layer / resolution / retrieval / migration / benchmarks). Only
  the *public ingestion API* (`Memory::add_episode`, `Memory::reextract_episodes`)
  is still gated by ISS-041 / ISS-042 (waiting on v03-resolution design).
- **conv-26 hit@5 progression**: 0% → 48% → **56%** in 4 days. Trajectory
  is good; remaining gap is concentrated in retrieval observability +
  migration wiring (ISS-044 / ISS-062), not in core algorithms.
- **Production fatal stubs**: 0. The `unimplemented!()` calls flagged in an
  earlier scan are all inside `#[cfg(test)]` mock impls, deliberately
  panicking on unused trait methods so accidental future call paths fail
  loudly. `Null*` types in `retrieval/plans/` are configuration choices
  (no-op rerankers, opt-out classifiers), not deferred work.
- **Ship distance**: ~3 P1 issues for *internal ship* (benchmark passes),
  ~5 P1 issues + a design doc for *external ship* (public API complete).

Same 25 queries, same anonymized substrate, different commits:

| Run | Date | Commit | hit@5 | Outcome shape | Source issue |
|---|---|---|---|---|---|
| RUN-0000 | 2026-04-26 | `0fe6156` | **0 / 25 (0%)** | All silent `ok` | discovered ISS-044 (backfill not wired) |

| RUN-0002 | 2026-04-28 | `602ed91` | **14 / 25 (56%)** | 2 terminal `EmptyResultSet`, no more silent zeros | ISS-063 fallback contract |

**Reading the curve:**

- The 0 → 48% jump (Apr 26 → 27) was unblocking the ingest path
  (ISS-044 wiring) — i.e. data finally reached the graph store at all.
- The 48% → 56% jump (Apr 27 → 28) was making fallback **explicit**:
  every primary plan now either contributes hits via Associative fallback
  or surfaces a terminal `EmptyResultSet`. No more "succeeded with zero
  results" lies.
- Remaining 11 misses split between: (a) genuinely missing seeds (graph
  signal too weak), (b) namespace mismatches silently swallowed
  (ISS-064), (c) downgrade-chain edge cases (ISS-059 in_review).

## Recalibration vs earlier reports

Two correction points worth recording, since I produced wrong numbers
twice during this audit and only caught it on cross-check:

1. **"56 / 39 stub markers" was wrong.** The grep counted
   `unimplemented!()` calls regardless of `cfg(test)`. Manually opening
   each location showed every one is a defensive `MockGraphStore` /
   `StubGraph` / `FakeGraph` panicking on unused trait methods —
   intentional, well-commented, protective. Production fatal stubs: 0.

2. **"5 Null* implementations to fill" was wrong.** Of the 10 `Null*`
   types in `retrieval/`:
   - `NullReranker`, `NullEntityLookup`, `NullLlmClassifier` are
     legitimate "no-op" configuration variants (the user opts out of
     reranking / LLM classification).
   - The rest are also legitimate configuration choices for plans where
     a particular sub-recaller isn't relevant. They're not "stubs to
     fill", they're *the disabled state*.

The cite-before-claim skill is doing real work here — the first answer
in each case felt confident and was specifically wrong on numbers.
This file is the cross-checked version.
